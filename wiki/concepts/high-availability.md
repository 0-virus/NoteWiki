# High Availability (고가용성)

## 한 줄 정의

**시스템의 일부 구성 요소가 죽어도 서비스 전체는 계속 동작하게 만드는 설계 원칙.**
"가용 시간 99.x%"의 그 가용성.

## 왜 필요한가

서버는 죽는다 — 하드웨어 고장, OS 패닉, 네트워크 단절, 메모리 누수, 배포 실수.
**"안 죽게"** 만드는 것은 불가능에 가깝지만, **"한 곳이 죽어도 서비스가 안 죽게"** 는 가능하다.
이게 HA의 핵심 발상이다.

## 핵심 개념: 단일 장애점 (SPOF)

**Single Point Of Failure** — 거기 하나만 죽어도 전체가 죽는 지점.
HA 설계는 **SPOF를 하나씩 없애는 작업**이다.

```
[클라이언트] → [서버] → [DB]
                ▲         ▲
              SPOF      SPOF
```

서버가 한 대뿐이면 그 서버가 SPOF. DB가 한 대뿐이면 DB가 SPOF.
서버 앞에 [[load-balancer|LB]]를 두면? **이번엔 LB가 SPOF**가 된다.
**SPOF를 없애는 작업은 보통 새로운 SPOF를 만든다 — 그래서 다중화는 연쇄적**이다.

## SPOF를 없애는 일반 패턴: 다중화

같은 역할의 컴포넌트를 둘 이상 두고 한쪽이 죽으면 다른 쪽이 받게 한다.
가장 자주 쓰이는 두 패턴:

### Active-Active

```
        [클라이언트]
          │
     ┌────┴────┐
     ▼         ▼
   [LB A]    [LB B]    ← 둘 다 항상 일하는 중
     │         │
     └────┬────┘
          ▼
       [서버들]
```

- 둘 다 동시에 트래픽을 처리한다.
- 한쪽이 죽으면 **즉시** 다른 쪽이 전부 받음 (페일오버 시간 ≈ 0).
- 자원이 항상 두 배로 가동 → **비용↑, 성능 여유↑**.

### Active-Passive

```
        [클라이언트]
          │
          ▼
       [LB A]   ← 일하는 중
       [LB B]   ← 대기 (활성화 안 됨)

       장애 시:
        [클라이언트]
          │
          ▼
       [LB B]   ← 새로 활성화
       [LB A]   ← 죽음
```

- 한 대만 실제로 일한다. 다른 대는 헬스체크하며 대기.
- 장애 감지 후 **활성화 시간(페일오버 지연)** 이 존재 — 보통 수 초~수십 초.
- 평소 한 대만 쓰니 **비용↓**, 대신 페일오버 동안 잠깐의 다운타임이 가능.

| 비교        | Active-Active        | Active-Passive       |
| ----------- | -------------------- | -------------------- |
| 평상시 자원 | 둘 다 사용           | 절반 놀고 있음       |
| 페일오버    | 즉시 (≈ 0)           | 수 초~수십 초        |
| 비용        | 더 비쌈              | 절약                 |
| 적합        | 무중단이 절대 조건   | 짧은 중단 허용 가능  |

영상의 마무리도 이 두 가지였다.

## HA가 사주는 두 가지

1. **수평 확장 ([[horizontal-vs-vertical-scaling]])** — 서버를 늘리는 결정이
   부수적으로 HA를 같이 사준다.
2. **무중단 배포** — 절반을 새 버전으로 교체하고, 끝나면 나머지를 교체하는 식.

반대로 **[[horizontal-vs-vertical-scaling|Vertical Scaling]]만 하는 시스템은 HA가 없다** —
한 대뿐이니까.

## "99.x% 가용성"의 의미

| 가용성 표기  | 연간 허용 다운타임 |
| ------------ | ------------------ |
| 99%          | 약 3.65일          |
| 99.9%        | 약 8.77시간        |
| 99.99%       | 약 52.6분          |
| 99.999%      | 약 5.26분          |

9가 하나 늘 때마다 비용은 보통 자릿수로 늘어난다.
**모든 시스템이 99.999%를 노릴 필요는 없다** — 비즈니스 요구에 맞춘다.

## 영균의 학습 맥락

학습 단계의 단일 Tomcat에는 HA가 없다. 그러나 이 개념을 알고 있어야:
- 부트캠프 프로젝트 발표 때 **다른 팀이 LB·다중화를 왜 그렸는지** 이해할 수 있다
- 면접에서 **"이 시스템의 SPOF는 무엇이냐"** 같은 질문에 대응할 수 있다

영균이 이번 인제스트로 잡고 싶었던 "다른 팀이 말한 그 인프라 이야기"의 정체가 결국 이 페이지다.
