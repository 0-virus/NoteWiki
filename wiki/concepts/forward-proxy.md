# Forward Proxy

## 한 줄 정의

**클라이언트 쪽에 위치한 [[proxy]].** 클라이언트가 외부로 나갈 때 거치는 관문이며,
바깥 입장에서는 프록시의 IP만 보인다.

## 왜 쓰는가 (4가지 동기)

1. **익명성 / 보안** — 클라이언트의 진짜 IP를 외부에 노출하지 않는다.
   서버 로그에는 프록시 IP가 찍힌다.
2. **접근 제어 (콘텐츠 필터링)** — 특정 사이트·도메인·시간대로의 접근을 막거나 허용.
   - 예: 사내망에서 업무 외 사이트 차단
   - 예: 학교에서 유해 콘텐츠 차단
3. **캐싱** — 사내 직원들이 자주 보는 페이지를 프록시가 들고 있다가 바로 응답.
   외부 왕복을 줄이고 대역폭을 절약.
4. **모니터링 / 로깅** — 누가 언제 어디로 나갔는지 한 점에서 관찰 가능.
   감사·법적 요구 대응의 단일 지점.

## 전형적 사용처

- **학교·기업·공공기관 사내망** — 외부로 나가는 모든 트래픽이 거치는 게이트웨이
- **VPN** — 본인 IP를 숨기고 다른 지역의 IP로 외부에 접속
- **개인용 익명 프록시** — 지역 제한 우회

## 흐름

```
[클라이언트] ──▶ [Forward Proxy] ──▶ [인터넷의 서버]
   IP=A                                  요청자 IP=프록시 IP, 클라이언트 A는 안 보임
```

서버는 누가 진짜 요청했는지 알 수 없다 (X-Forwarded-For 같은 헤더가 없다면).

## [[reverse-proxy]]와의 결정적 차이

같은 원리지만 **누구를 숨기느냐가 정반대**:

| 질문                  | Forward Proxy            | Reverse Proxy             |
| --------------------- | ------------------------ | ------------------------- |
| 프록시는 어디에 있나? | 클라이언트 옆            | 서버 옆                   |
| 누구의 IP를 숨기나?   | 클라이언트               | 서버                      |
| 누가 보호받나?        | 클라이언트(나가는 쪽)    | 서버(받는 쪽)             |
| 클라이언트는 프록시를 의식하나? | **명시적으로 설정** (브라우저 프록시 설정) | **모름** — 그냥 서버 URL로 보냈다고 생각 |

마지막 행이 특히 중요하다: **Forward는 명시적, Reverse는 투명**.

## 영균의 학습 맥락

학습 단계에서는 직접 만들 일이 거의 없다. 다만 **"왜 회사 와이파이에서는
유튜브가 안 되는가"** 같은 일상 현상이 forward proxy 때문임을 알아두면,
나중에 사내 인프라 이야기를 들을 때 헷갈리지 않는다.
