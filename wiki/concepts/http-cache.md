# HTTP 캐시 (브라우저 캐시)

## 한 줄 정의

**"한 번 받은 리소스는 저장해뒀다 재사용한다"는 웹의 성능 최적화.** OS의 CPU 캐시와 같은 발상 — 느린 원본 접근(네트워크)을 빠른 임시 저장소(브라우저 캐시)로 대체한다.

## 왜 필요한가

같은 이미지(예: 1.1MB `star.jpg`)를 볼 때마다 서버에서 다시 받으면 **네트워크 시간·트래픽·서버 부하**가 낭비된다. 캐시하면:

- **클라이언트**: 네트워크 다운로드 회피 → 빠른 로딩
- **서버**: 네트워크 사용량 감소 → 비용 절감

하지만 캐시는 **양날의 검**이다. 잘못 캐싱하면 **낡은 리소스(stale)** 를 보여준다. 그래서 "얼마나 오래 유지할지"를 다루는 헤더 기술이 필요하다.

## 3단계 구조로 이해하기

캐시 전략은 3개 층으로 나뉜다. 각각 별도 페이지로 판다.

### 1. 유효 시간 지정 — [[cache-control]]

서버가 `Cache-Control: max-age=60` 을 주면, 브라우저는 60초 동안 **서버에 묻지도 않고** 캐시에서 바로 꺼내 쓴다(fresh 상태).

```
요청 → 서버가 max-age=60과 함께 응답 → 캐시에 60초 저장
재요청(60초 내) → 서버 안 감, 캐시에서 즉시 반환 ✅
재요청(60초 초과) → 캐시 만료(stale), 서버로 감 🚩
```

문제: 만료됐다고 무조건 새로 받는 건 낭비다. **내용이 안 바뀌었으면** 다시 받을 이유가 없다. → 2단계로.

### 2. 만료 후 검증 — [[conditional-request|조건부 요청]]

만료된 캐시라도 버리지 않고, "서버야, 이거 그대로야?"를 **가볍게 물어본다**. 안 바뀌었으면 서버가 `304 Not Modified`(본문 없음, ~0.1MB)로 답하고, 브라우저는 **기존 캐시를 재사용**한다. 검증 수단은 두 가지:

- 수정 시각 방식: `Last-Modified` ↔ `If-Modified-Since`
- 버전 해시 방식: [[etag]] ↔ `If-None-Match` (더 정밀)

### 3. 캐시 금지 — 무효화

민감하거나 항상 최신이어야 하는 데이터는 아예 캐시를 막거나 매번 검증하게 한다. `no-store`·`no-cache`·`must-revalidate` → [[cache-control]] 참고.

## [[proxy]] 캐시 — private vs public

캐시가 저장되는 위치는 둘로 나뉜다.

| 구분 | 위치 | Cache-Control |
| --- | --- | --- |
| **private 캐시** | 각 사용자의 브라우저 | `private`(기본) |
| **public 캐시** | 중간의 [[proxy]] 캐시 서버(CDN 등) | `public` |

원서버가 미국에 있고 사용자가 한국이면, 한국에 [[reverse-proxy|프록시 캐시 서버]](CDN)를 두어 **최초 1회만** 미국에서 받고 이후엔 국내에서 배포한다. 유튜브 고용량 영상이 빠른 이유. 프록시 전용 헤더:

- `Cache-Control: s-maxage` — 프록시 캐시에만 적용되는 max-age
- `Age: 60` — 원서버 응답이 프록시에 머문 시간(초)

## 왜 캐시가 [[cors]] 에러를 일으키기도 하나

브라우저는 이미지 등을 자동 캐시하는데, 캐시된 응답에 CORS 허용 헤더가 없던 버전이 남아 있으면 뒤늦게 CORS 에러가 튀어나올 수 있다. 캐시와 CORS가 얽히는 지점.

## 영균 학습 트리에서

- [[etag]]를 이미 정리해뒀다 — 그 ETag가 여기 2단계(조건부 요청)의 핵심 부품이다. 캐시 전체 그림 안에서 ETag의 위치를 잡는 게 이 페이지의 역할.
- [[preflight-request|예비 요청]] 캐싱(`Access-Control-Max-Age`)도 결국 이 캐시 메커니즘의 응용.
- Spring에서 정적 리소스에 `Cache-Control`을 걸거나, [[etag|ShallowEtagHeaderFilter]]로 조건부 요청을 켜는 게 실무 적용점.
