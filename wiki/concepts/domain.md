# Domain (도메인)

## 한 줄 정의

소프트웨어에서 "도메인"은 **두 가지 완전히 다른 맥락**으로 쓰인다.
- **비즈니스 도메인** — 이 소프트웨어가 다루는 **현실 세계의 영역**. (Spring·DDD 맥락의 그 도메인)
- **네트워크 도메인** — 인터넷 주소의 한 구역. (`google.com` 같은 것)

두 단어는 어원만 같을 뿐 서로 다른 개념이다. 이 페이지는 둘을 한 번에 정리해두는 분기 지점.

---

## 1. 비즈니스 도메인 — "이 소프트웨어가 사는 세계"

### 어원에서 출발

`domain`은 라틴어 `dominium`(영역, 지배권)에서 왔다. 일상 영어에서도 "전문 영역(area of expertise)"이라는 뜻으로 쓰인다 — *"That's not my domain."*

소프트웨어에 그대로 가져오면: **이 프로그램이 해결하려는 문제의 영역**.

### 구체적 예시

쇼핑몰을 만든다고 하자. 이 쇼핑몰의 "도메인"은 **온라인 쇼핑이라는 현실 세계의 영역**이다. 그 세계에는 이런 개념들이 산다:

```
[쇼핑 도메인]
  - 회원 (Member)
  - 상품 (Product)
  - 장바구니 (Cart)
  - 주문 (Order)
  - 결제 (Payment)
  - 배송 (Delivery)
  - 리뷰 (Review)

각 개념에 딸린 규칙:
  - "배송 완료된 주문은 취소할 수 없다"
  - "재고 0인 상품은 장바구니에 담을 수 없다"
  - "쿠폰은 한 주문에 하나만 적용된다"
```

이 **개념들 + 그들 사이의 규칙**이 곧 "쇼핑 도메인"이다.

병원 예약 시스템이면 도메인이 다르다 — 환자, 의사, 진료, 예약, 처방. 거기엔 "이미 예약된 시간엔 다른 환자를 못 받는다" 같은 규칙이 있다.

**즉, 도메인은 "코드가 아니라 현실"에 먼저 존재한다.** 프로그래머가 그걸 코드로 옮기는 거다.

### 도메인 모델 — 도메인을 코드로 옮긴 것

도메인이 "현실 세계의 영역"이라면, **도메인 모델**은 그 영역을 코드로 표현한 객체 묶음이다. [[three-tier-architecture]]에서 표로 등장했던 그 "Domain Model".

```java
// 쇼핑 도메인의 모델 (Java 코드로)
public class Order {            // ← 도메인 개념 "주문"
    private Long id;
    private OrderStatus status;
    private List<OrderItem> items;

    public void cancel() {       // ← 도메인 규칙
        if (status == OrderStatus.SHIPPED) {
            throw new IllegalStateException("배송 완료된 주문은 취소할 수 없습니다.");
        }
        this.status = OrderStatus.CANCELED;
    }
}
```

`Order` 클래스 안에 **현실 세계의 "주문"이라는 개념과 규칙**이 그대로 살아 있다. 이게 도메인 모델.

### 엔티티 = 도메인 모델의 핵심 시민

[[dto-vs-entity]]에서 말한 "엔티티는 도메인의 본체"의 의미는 이 그림에서 드러난다.

```
도메인         = 쇼핑이라는 현실 세계
  └─ 도메인 모델  = 그 세계를 코드로 옮긴 객체들의 묶음
       └─ 엔티티     = 도메인 모델 중 식별자(id)를 가진 핵심 객체
                       (Order, Member, Product 같은 것)
```

"엔티티가 비즈니스 로직과 직결된다"는 말은 곧 **"엔티티는 도메인의 규칙을 자기 안에 품고 있어야 한다"**는 뜻이다. `Order.cancel()`이 "배송 완료된 주문은 취소 불가"라는 도메인 규칙을 알고 있는 것처럼.

### DO · Entity · VO — 도메인 모델 안의 세 카테고리

도메인 모델에 속하는 객체를 통칭하여 **DO (Domain Object)** 라고 부른다. DDD 관점에서 DO는 다시 두 가지로 갈린다.

```
DO (Domain Object) — 도메인 모델의 객체 일반
   ├── Entity        : "식별자(id)로 구별되는" 객체
   │                   같은 필드라도 id가 다르면 다른 객체.
   │                   예: Order (주문 #42 ≠ 주문 #43)
   │
   └── Value Object  : "값으로 구별되는" 객체. id 없음.
                       같은 값이면 같은 객체로 취급.
                       예: Money(1000, KRW), Address("서울시 강남구 ...")
```

- **Entity**는 "이 주문은 다른 주문과 다른 주문이다"라는 식별이 중요한 객체.
- **Value Object**는 "1000원짜리 두 개는 같은 1000원이다"라는 값 동등성이 중요한 객체.
  → [[equals-hashcode]]를 오버라이딩해야 하는 가장 전형적인 사례가 VO.

> ⚠️ **한국 실무 함정**: "DO"라는 용어가 잘 쓰이지 않고, 쓰여도 사람마다 정의가 다르다.
> - DDD 책: 위 위계대로 "DO ⊃ Entity ∪ VO".
> - 일부 한국 코드베이스: "DO = Entity와 사실상 같은 뜻"으로 씀.
> - 다른 한국 코드베이스: "DO = DB 테이블과 매핑되는 객체"로 좁혀 씀 (Entity와 거의 같지만 JPA 어노테이션 없는 버전).
> 팀에서 "DO"라는 용어가 등장하면 **합의된 의미를 먼저 확인**할 것.

### 영균 실무 단계에서

처음엔 "**DTO ↔ Entity**" 두 개로만 생각해도 충분. DO/VO를 의식해야 할 때는:
- DDD 책이나 자료를 읽을 때
- `Money`·`Address`·`Period` 같은 Value Object를 도입하기 시작할 때
- 외국(특히 일본·중국) 코드베이스에서 `~DO` 접미사를 마주칠 때

### 왜 "비즈니스 로직"이 아니라 "도메인"이라고 부르나

영균이 원리를 묻는 사람이니 한 걸음 더. 왜 굳이 "도메인"이라는 단어를 쓸까?

**관점이 다르기 때문이다.**

- "비즈니스 로직"은 *코드 관점*의 단어 — "이 로직은 비즈니스에 속한다".
- "도메인"은 *문제 관점*의 단어 — "이 소프트웨어가 다루는 **현실 영역 그 자체**".

도메인이라는 단어를 쓰면 자연스럽게 이런 질문이 따라온다:
- "쇼핑 도메인 전문가(예: 실제 쇼핑몰 운영자)에게 물어보면 뭐라고 답할까?"
- "이 코드는 도메인 전문가가 이해할 수 있는 언어로 쓰여 있나?"

이게 **Domain-Driven Design (DDD)** 의 출발점이다. 코드를 비즈니스의 언어로 쓰자는 사조. 영균이 지금 DDD를 깊이 팔 필요는 없지만, "도메인"이라는 단어가 그 사조에서 왔다는 걸 알면 왜 모두가 이 단어를 쓰는지 이해된다.

### 위키 안에서 "도메인"이 등장하는 곳

- [[dto-vs-entity]]: "Entity는 비즈니스 도메인의 본체"
- [[three-tier-architecture]]: "Domain Model은 비즈니스 규칙을 실행하는 진짜 객체"
- "도메인 변경이 클라이언트로 새어 나간다" → "현실 세계의 규칙이 바뀌어 코드를 고쳤는데, API 응답까지 같이 바뀐다"

## 2. 네트워크 도메인 — "인터넷 주소의 한 구역"

같은 단어지만 완전히 다른 세계. 위키에선 [[proxy]] 계열 페이지에서 등장한다.

### 정의

인터넷에서 **사람이 읽을 수 있는 주소의 한 구역**. IP 주소(`142.250.207.78`)는 외우기 어려우니, 사람이 읽을 수 있는 이름(`google.com`)을 붙인 것.

```
naver.com         ← 도메인
shop.naver.com    ← 서브도메인 (shop이 naver.com 아래에 있음)
www.naver.com     ← 서브도메인 (www도 마찬가지)
```

도메인 이름을 실제 IP로 바꿔주는 시스템이 **DNS(Domain Name System)**.

### 위키 안에서 "도메인"이 이 뜻으로 쓰인 곳

- [[forward-proxy]] · [[proxy]]에서 "특정 도메인 차단" 같은 표현.
- "회사 네트워크에서 youtube.com 도메인을 막는다" 같은 시나리오.

## 비교 정리

| 구분 | 비즈니스 도메인 | 네트워크 도메인 |
| --- | --- | --- |
| 무엇 | 현실 세계의 문제 영역 | 인터넷 주소의 한 구역 |
| 예시 | 쇼핑, 게시판, 예약 | `google.com`, `naver.com` |
| 어디서 만남 | [[dto-vs-entity]], [[three-tier-architecture]], Spring·DDD | [[proxy]], [[forward-proxy]], DNS |
| 어원 | 라틴어 *dominium*(영역) — 둘 다 여기서 옴 | 같음 |

**같은 어원에서 출발해 "내가 관할하는 영역"이라는 뜻을 공유**하지만, 한쪽은 *현실 문제의 영역*, 다른 쪽은 *인터넷 주소의 영역*으로 갈라졌다. 둘을 헷갈리지 않게 항상 **문맥을 먼저** 보자.

## 한 줄 요약

> **비즈니스 도메인 = 이 소프트웨어가 다루는 현실 세계의 영역.**
> **엔티티 = 그 세계의 규칙과 정체성을 코드로 품은 객체.**
> 그래서 "엔티티는 도메인의 본체"이고, "비즈니스 로직은 엔티티 안에 살아야 한다"가 같은 말이 된다.
