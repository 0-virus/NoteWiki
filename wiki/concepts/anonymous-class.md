# 익명 클래스 (Anonymous Class)

## 한 줄 정의

이름 없이 **인터페이스 구현체(또는 추상 클래스 하위 클래스)를 즉석에서 만들어
변수에 담는** 문법. [[lambda]]의 전신이다.

```java
IAdd<Integer> a = new IAdd<Integer>() {
    @Override
    public Integer add(Integer x, Integer y) { return x + y; }
};
```

`new 인터페이스() { ... }` — 원래 인터페이스는 `new` 할 수 없지만, 중괄호 안에
구현을 바로 적어 "이름 없는 구현 클래스 + 그 인스턴스"를 한 번에 만든다.

## 왜 쓰는가

한 번만 쓸 구현체를 위해 별도 `.java` 파일/클래스를 만드는 건 번거롭다. 익명
클래스는 **그 자리에서 일회용 구현**을 끼워 넣는다. 이벤트 핸들러, 콜백 등에 흔했다.

## 람다와의 차이 (왜 람다가 따로 생겼나)

함수형 인터페이스(추상 메서드 1개)를 익명 클래스로 구현하면 보일러플레이트가 많다
(`new ...() { @Override public ... { } }`). 람다는 그 군더더기를 걷어낸 축약 문법이다.
하지만 둘은 **같지 않다**:

| 구분           | 익명 클래스                    | 람다                          |
| -------------- | ------------------------------ | ----------------------------- |
| `this`         | 자기 자신(익명 클래스 인스턴스) | 바깥 클래스 인스턴스          |
| 스코프         | 독립 클래스 스코프 — 필드 선언 가능 | 바깥 메서드의 지역 변수 스코프 공유 |
| 적용 대상      | 메서드 여러 개 인터페이스, 추상 클래스 상속도 가능 | 함수형 인터페이스(추상 메서드 1개)만 |
| 내부 처리      | 컴파일 시 별도 클래스 파일 생성 | JVM `invokedynamic` 최적화    |

> 핵심: 익명 클래스가 람다보다 **더 넓다.** 메서드가 2개 이상인 인터페이스나 추상
> 클래스를 즉석 구현해야 하면 람다는 못 쓰고 익명 클래스를 써야 한다. 추상 메서드
> 1개짜리라면 람다가 더 간결하다. `this`·스코프 차이는 [[lambda]] 참고.
