# 동시성 문제 (Race Condition)

## 한 줄 정의

**공유 자원에 여러 주체가 동시에 접근**할 때, 작업이 엇갈려 값이 꼬이는 문제.

## 메커니즘

`count++`는 사실 "읽기 → 더하기 → 쓰기"의 3단계다. 두 스레드가 끼어들면:

```
스레드 A: count 읽음 (0)
스레드 B: count 읽음 (0)    ← A가 아직 저장 안 했는데 B도 읽어버림
스레드 A: 0 + 1 = 1 저장
스레드 B: 0 + 1 = 1 저장    ← 2가 되어야 하는데 1
```

둘 다 1을 저장해 카운트가 어긋난다. 이것이 **Race Condition**.

## 두 층위에서 같은 문제

이 소스가 강조하는 핵심 연결 — **근본 원인은 하나, 다루는 층이 다르다.**

| | Servlet/Java (메모리 레벨) | DB 트랜잭션 (DB 레벨) |
| --- | --- | --- |
| 공유 자원 | 인스턴스 변수 (힙) | 같은 row |
| 충돌 주체 | 여러 스레드 | 여러 트랜잭션 |
| 해결 | 상태를 인스턴스 변수로 안 둠 | 격리 수준, 락(lock) |

> 근본 원인: **공유 자원에 여러 주체가 동시에 접근할 때 생기는 문제.**
> [[servlet-lifecycle]]의 싱글톤 동시성은 *애플리케이션 레벨*,
> 트랜잭션 격리 수준은 *DB 레벨* — 같은 문제를 다른 층에서 다룬다.

## Servlet에서의 해결

[[servlet]]은 싱글톤이라 인스턴스가 공유된다. 그래서:

- **인스턴스 변수** → 힙에 하나, 모든 스레드 공유 → 위험
- **지역 변수** → 각 스레드의 스택에 독립 생성 → 안전

→ 원칙: **Servlet/[[spring-bean]] 안에 상태를 인스턴스 변수로 두지 마라.**

## 영균의 학습 맥락

이전에 Java 프로젝트에서 공부한 **트랜잭션 격리 수준**이 DB 레벨의 동시성 제어다.
이번 Servlet 싱글톤 이야기는 애플리케이션 레벨의 동시성 제어 — 두 지식이 한 가지에서 만난다.
