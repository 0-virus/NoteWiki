# Proxy (프록시)

## 한 줄 정의

**네트워크 통신의 중간자.** 두 엔티티 사이에 끼어 통신을 중계하면서,
한쪽의 정체를 감추거나, 통신 내용을 제어·기록·캐싱한다.

## 왜 필요한가

직접 연결은 단순하지만 **무방비**다. 중간에 한 단계만 끼워 넣으면 세 가지를 얻는다.

1. **익명화** — 뒤에 있는 쪽의 IP가 외부에 노출되지 않는다.
2. **제어** — 어떤 트래픽을 막을지 / 어디로 보낼지 결정할 수 있는 게이트.
3. **최적화** — 자주 같은 응답이라면 캐싱해서 뒤쪽을 건드리지 않아도 된다.

이 세 효용이 클라이언트 쪽이냐 서버 쪽이냐에 따라 [[forward-proxy]] / [[reverse-proxy]]로 갈린다.

## Forward vs Reverse — 같은 원리, 반대 방향

```
[Forward Proxy]                    [Reverse Proxy]
클라이언트 ─▶ 프록시 ─▶ 인터넷    클라이언트 ─▶ 프록시 ─▶ 서버들
            (클라이언트 쪽)                       (서버 쪽)
숨기는 것: 클라이언트 IP          숨기는 것: 서버 IP
보호 대상: 클라이언트             보호 대상: 서버
```

핵심 통찰: **둘은 위치만 다른 한 쌍**이다. "프록시 = 한 종류"라는 오해가 가장 흔하다.

| 구분        | Forward Proxy             | Reverse Proxy             |
| ----------- | ------------------------- | ------------------------- |
| 위치        | 클라이언트 측 (사내망 등) | 서버 측 (앞단)            |
| 숨기는 IP   | 클라이언트                | 서버                      |
| 주 사용처   | 사내망 게이트웨이, VPN    | 웹 서비스 앞단            |
| 대표 시나리오 | 콘텐츠 필터링·모니터링    | 로드 밸런싱·DDoS 방어     |
| 대표 도구   | Squid                     | [[nginx]], HAProxy        |

## 공통으로 제공하는 기능

- **IP 익명화** — 뒤쪽 정체를 숨김
- **접근 제어** — 특정 IP/도메인 차단·허용
- **캐싱** — 자주 쓰이는 응답을 프록시에 저장 → 백엔드 부담↓, 응답 지연↓
- **모니터링/로깅** — 모든 트래픽이 한 점을 거치므로 관찰이 쉬움

## 영균의 학습 맥락

[[servlet-container]]에 띄운 Tomcat이 외부에 직접 노출되면 IP·포트가 그대로 공개되고,
공격·과부하·SSL 같은 부담을 Tomcat이 다 떠안는다.
앞에 [[reverse-proxy]] 한 대(보통 [[nginx]])를 두는 순간:
- 외부에는 Nginx만 보임 (Tomcat 위치 은닉)
- HTTPS 종단·정적 파일 서빙·캐싱을 Nginx가 가져감 → Tomcat은 동적 처리에 집중
- Tomcat을 여러 대로 늘리면 자연스럽게 [[load-balancer]]가 된다

이게 [[scaling-a-web-app]] 흐름의 첫 단계다.
