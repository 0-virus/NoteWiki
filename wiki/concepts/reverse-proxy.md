# Reverse Proxy

## 한 줄 정의

**서버 쪽에 위치한 [[proxy]].** 외부 요청을 먼저 받아 뒤쪽 실제 서버(들)로 넘겨주며,
바깥에서 봤을 때는 프록시만 보인다.

## 왜 쓰는가

직접 [[servlet-container]](Tomcat) 한 대를 외부 80/443 포트에 노출하면 다음 부담을 모두
Tomcat이 떠안는다: HTTPS 종단, 정적 파일 서빙, 봇·DDoS 방어, 여러 인스턴스 분산.
이걸 **앞단 한 대로 분리**한 것이 리버스 프록시다.

### 4대 효용

1. **서버 IP/위치 은닉** — 외부는 프록시 IP만 안다. 실제 서버는 사설 IP로 숨김 →
   서버를 직접 노린 공격이 어려움. **[[high-availability]]의 첫 단계.**
2. **[[load-balancer|로드 밸런싱]]** — 뒤에 서버를 여러 대 두고 트래픽을 분산.
   리버스 프록시의 **가장 흔한 활용**.
3. **캐싱 / 정적 파일 처리** — 이미지·CSS·JS는 프록시가 직접 응답.
   Tomcat은 동적 처리에만 집중 → 백엔드 부담 감소.
4. **TLS(HTTPS) 종단** — 인증서 관리·암호화 해제를 프록시에서 처리하고
   백엔드는 평문 HTTP로 받는다 → 인증서를 한 곳에서만 관리.

## 흐름

```
        [클라이언트들]
              │
              ▼
       [Reverse Proxy]   ← 외부에 보이는 IP
       (Nginx 등)
        │   │   │
        ▼   ▼   ▼
    [Tomcat1][Tomcat2][Tomcat3]   ← 사설 IP, 외부에서 직접 접근 불가
```

클라이언트는 자신이 단일 서버와 통신한다고 믿지만, 실제로는 여러 서버가 뒤에 있다.
**투명성**이 forward proxy와 갈리는 결정적 지점 ([[forward-proxy]] 참고).

## 대표 구현

- **[[nginx]]** — 가장 흔한 선택. 단일 바이너리·이벤트 기반·메모리 효율적.
- **HAProxy** — 더 풍부한 LB 알고리즘, L4/L7 모두 강함.
- **AWS ALB / GCP Load Balancer** — 클라우드 매니지드 리버스 프록시.

## DDoS·공격 방어 관점

리버스 프록시가 자동으로 DDoS를 막아주는 것은 아니지만:
- **서버 위치 은닉**으로 직접 공격을 어렵게 만든다
- 프록시 자체에 **레이트 리미트·IP 차단** 등의 1차 방어선을 둘 수 있다
- 프록시는 보통 정적 리소스 응답이 빨라 **공격 흡수력**이 백엔드보다 높다

## 영균의 학습 맥락

지금까지 [[servlet-container]]에서 Tomcat을 **단독으로** 띄워 보았다.
이 상태가 **외부에 Tomcat이 직접 보이는 구조**다.
리버스 프록시는 그 앞에 한 층을 더 두는 것 — 학습 트리의 자연스러운 다음 단계지만,
영균 본인은 **시스템 디자인 카테고리**의 독립된 가지로 다루기로 결정했다.

실습 진입은 [[nginx]] 페이지의 리버스 프록시 설정 예시로 시작하면 된다.
