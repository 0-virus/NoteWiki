# Load Balancing Algorithm (분배 알고리즘)

## 한 줄 정의

[[load-balancer]]가 **다음 요청을 어느 서버로 보낼지 정하는 규칙**.
이 규칙이 LB의 성격을 결정한다.

## 왜 알고리즘이 여러 개인가

"골고루 나눠주면 되지" 같지만 현실은 다르다:
- 서버마다 **성능이 다를 수 있다** (스펙·세대 차이)
- 요청마다 **처리 시간이 다르다** (가벼운 GET vs 무거운 집계)
- 어떤 요청은 **이전 요청과 같은 서버**로 가야 한다 (세션 유지)

이 세 변수에 대응하려고 알고리즘이 분화됐다.

## 4가지 핵심 알고리즘

### 1. Round Robin (RR)

**순차 분배.** 서버1 → 서버2 → 서버3 → 서버1 → ... 순환.

```
요청 1 ─▶ 서버1
요청 2 ─▶ 서버2
요청 3 ─▶ 서버3
요청 4 ─▶ 서버1   (다시 처음으로)
```

- **장점**: 구현이 가장 단순, 모든 서버의 사양이 동일할 때 균등.
- **단점**: 서버 성능 차이를 무시한다. 빠른 서버의 여력을 낭비.
- **언제**: 서버군이 동일 스펙으로 구성됐을 때.

### 2. Weighted Round Robin (WRR)

**가중치 있는 순차 분배.** 성능 좋은 서버에 더 자주.

```
가중치: 서버1=2, 서버2=1, 서버3=1
요청 1,2 ─▶ 서버1   (가중치 2 → 두 번)
요청 3   ─▶ 서버2
요청 4   ─▶ 서버3
요청 5,6 ─▶ 서버1
...
```

- **장점**: 이기종 서버군에서 효율적.
- **단점**: 여전히 **실시간 부하**는 못 본다 — 가중치는 고정값.
- **언제**: 서버 사양이 섞여 있을 때.

### 3. Least Connection

**현재 연결 수가 가장 적은 서버에 보낸다.** 실시간 부하 반응.

```
현재 상태: 서버1=5연결, 서버2=2연결, 서버3=3연결
새 요청 ─▶ 서버2   (가장 한가함)
```

- **장점**: 응답이 오래 걸리는 서버를 자동으로 피해간다.
  (응답 시간이 길면 연결이 누적되니까)
- **단점**: 연결 수를 추적해야 함 → LB의 상태 관리 부담.
- **언제**: 요청 처리 시간이 들쭉날쭉할 때 (DB 집계·파일 업로드 등).

### 4. IP Hash

**클라이언트 IP를 해시해서 항상 같은 서버로 보낸다.**

```
hash("192.168.1.10") % 3 = 0 → 항상 서버1
hash("192.168.1.20") % 3 = 2 → 항상 서버3
```

- **장점**: **세션 유지(Sticky Session)**.
  세션 데이터를 서버 메모리에 보관하는 구조라면 같은 클라이언트는 같은 서버로 가야 한다.
- **단점**:
  - 특정 IP군이 한 서버에 몰리면 불균형
  - 서버 추가/제거 시 해시가 재배치되어 기존 세션이 깨질 수 있음 (consistent hashing으로 완화)
- **언제**: 세션을 외부 저장소(Redis)로 빼지 않은 단계에서 임시방편으로.

## 비교표

| 알고리즘            | 본다           | 강점             | 약점               |
| ------------------- | -------------- | ---------------- | ------------------ |
| Round Robin         | 순서만         | 단순             | 성능 차이 무시     |
| Weighted RR         | 고정 가중치    | 이기종 대응      | 실시간 부하 무시   |
| Least Connection    | 현재 연결 수   | 부하 실시간 반응 | 상태 추적 필요     |
| IP Hash             | 클라이언트 IP  | 세션 어피니티    | 분포가 치우칠 위험 |

## "어떤 걸 골라야 하나" — 의사결정

```
서버 사양이 동일?
├─ Yes → 요청 처리 시간이 일정?
│         ├─ Yes → Round Robin
│         └─ No  → Least Connection
└─ No  → 세션 유지 필요?
          ├─ Yes → IP Hash
          └─ No  → Weighted Round Robin
```

> ⚠️ 단순화된 의사결정 트리다. 실무에선 알고리즘을 조합하거나 (예: WRR + 헬스체크) 자체 가중치를 둔다.

## 영균의 학습 맥락

세션 유지 이야기는 [[servlet-container]] 학습에서 자연스럽게 만나게 된다 —
Tomcat의 `HttpSession`이 서버 메모리에 살기 때문이다. 그래서:
- **단순 분산**만 필요하면 Round Robin
- **로그인 세션이 끊기면 안 되면** IP Hash 또는 외부 세션 스토어(Redis)
가 학습 단계에서 마주칠 첫 갈래길이다.

실습은 [[nginx]]의 `upstream` 블록에서 `least_conn`·`ip_hash` 키워드를 바꿔보는 것으로 시작.
