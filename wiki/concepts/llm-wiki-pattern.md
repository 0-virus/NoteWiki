# LLM Wiki Pattern

## 한 줄 정의

원본 문서를 chunk로 검색([[rag]])하는 대신, **AI가 지속적으로 컴파일·유지하는 위키**를 만들어 그 위에서 답변하는 패턴. Andrej Karpathy가 제안.

> **이 저장소(NoteWiki)가 따르는 패턴이 바로 이것이다.**

## 왜 등장했나 — RAG의 한계

[[rag]]는 강력하지만 본질적 약점이 있다:

1. **chunk 간 연결이 끊긴다** — 문서를 잘게 쪼개면서 전체 맥락·개념 간 연결이 사라진다.
2. **매 질문마다 매번 검색** — 같은 질문 → 같은 검색 → 같은 사고가 반복된다. 결과가 누적되지 않는다.
3. **chunk 품질이 답변 품질** — 검색이 실패하면 답도 실패한다.

→ 사람은 위키(Wikipedia)를 만들어 지식을 누적해왔다. **LLM도 그렇게 하면 안 될까?**

## 핵심 아이디어

원본은 그대로 두고, AI가 그 위에 **구조화된 위키**를 만든다:

- 개념 페이지(`concepts/`) — 하나의 개념 = 하나의 파일
- 흐름 페이지(`flows/`) — 개념들의 활용 순서·진화
- 소스 페이지(`sources/`) — 원본 요약·출처
- 색인(`index.md`) — 카탈로그
- 로그(`log.md`) — 시간순 작업 기록

페이지들은 **`[[wikilink]]`로 상호 연결**된다 — 이게 RAG의 chunk와 결정적으로 다른 점.

## 3대 작업

| 작업 | 트리거 | 동작 |
| --- | --- | --- |
| **Ingest** | 원본 추가 | AI가 원본을 읽고 위키 페이지를 만들거나 갱신 |
| **Query** | 사용자 질문 | 위키에서 답하고, 가치 있는 답변은 새 페이지로 환류 |
| **Lint** | 점검 지시 | 위키 전체 모순·고아 페이지·갱신 누락 점검 |

→ 질문조차도 **위키를 키운다**. 채팅 기록으로 사라지지 않는다.

## RAG와의 비교

| | [[rag]] | LLM Wiki Pattern |
| --- | --- | --- |
| 자료 형태 | chunk 단위 (단편적) | 페이지·연결 (구조적) |
| 갱신 시점 | 인덱싱 시 자동 | AI가 ingest 때 컴파일 |
| 검색 비용 | 매 질문마다 | 위키 index 한 번 |
| 연결성 | chunk 간 끊김 | wikilink로 보존 |
| 누적성 | 없음 (질문은 휘발) | 있음 (질문도 페이지로 환류) |
| 자료 규모 | 무제한 | 컴파일 가능한 범위 |
| 적합한 케이스 | 대규모·자주 갱신 | **정제된 지식·연결성 중요** |

→ 자세한 트레이드오프는 [[rag-vs-llm-wiki]] 참조.

## 왜 NoteWiki에 적합한가

박영균의 비전은 "**자주 쓰는 개념과 연결 관계를 기록해, 맥락에 맞춰 꺼내주는 AI 색인 도구**"다. 여기서 핵심은:

- 자료 규모: 무한정 늘어나지 않음 (개인 학습·실무 노트)
- 가장 중요한 자산: **개념 간 연결** (예: Tomcat → Servlet → Spring MVC)
- 누적성: 과거의 나도 미래의 나도 같은 위키를 본다

이 세 가지가 모두 RAG의 약점이자 Wiki Pattern의 강점에 해당한다 → **Wiki Pattern이 옳은 선택**.

## 한계

- **자료가 방대해지면 컴파일 부담** — 위키를 매번 직접 갱신하기 어려워짐
- **자동 갱신이 어려움** — 원본이 빠르게 바뀌면 위키가 따라가지 못함
- **AI 의존도가 높음** — 위키의 품질이 컴파일하는 AI의 품질에 좌우됨

→ 자료가 충분히 커지면 **두 패턴을 합칠 수 있다**: 위키는 위키대로, 원본은 [[rag]]로 검색해 보완.
