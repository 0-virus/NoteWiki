# Session 인증

## 왜 등장했나

[[cookie]] 방식의 근본 문제는 **민감한 정보가 클라이언트에 그대로 저장**된다는 것.
세션 방식은 정보를 서버에 두고, 클라이언트에는 "열쇠(Session ID)"만 준다.

## 인증 흐름

```
1. 로그인 요청
2. 서버: { session_id: "abc123", user_id: 42, role: "admin" } → 서버 메모리/DB에 저장
3. 클라이언트: 쿠키에 session_id=abc123 만 저장 (민감 정보 없음)
4. 이후 요청 시 session_id 쿠키 첨부
5. 서버: "abc123... 메모리 뒤져봐야지" → user_id 42, admin 확인 → 인가
```

## Stateful — 핵심 특성

세션 방식은 **Stateful**: 서버가 "abc123은 user 42번" 이라는 사실을 **기억(state)** 하고 있어야 한다.

→ 서버를 2대로 늘리면? 각 서버가 서로 다른 세션 메모리를 가짐 → **세션 공유 문제**
→ 서버 재시작 시 세션 날아감 → 로그아웃

## 세션 ID 탈취 + IP 바인딩 방어

세션 ID는 단순 문자열 → 탈취하면 서버는 진짜 사용자와 구별 불가.

**IP 바인딩**: 로그인 시 세션에 클라이언트 IP도 함께 저장, 이후 요청의 IP와 대조.

```
세션에 저장: { session_id: "abc123", user_id: 42, ip: "203.0.113.10" }
요청 도착 → 실제 IP == 저장된 IP? → 불일치 시 세션 무효화
```

**한계**:

| 상황 | 문제 |
|------|------|
| 공격자가 같은 NAT/VPN 안 | IP가 동일하게 보임 → 방어 무력화 |
| 모바일 → WiFi 전환 | 정상 사용자 IP 변경 → 강제 로그아웃 |
| 기업 내부망 | 여러 사용자가 같은 출구 IP 공유 |

실무에서는 단독 사용보다 **IP + User-Agent 조합**, **HTTPS**, **짧은 세션 만료** 함께 사용.

## Cookie vs Session 비교

| | Cookie | Session |
|-|--------|---------|
| 저장 위치 | 클라이언트 | 서버 |
| 저장 형식 | 텍스트 | Object |
| 만료 시점 | 저장 시 설정 | 브라우저 종료 시 |
| 용량 제한 | 쿠키당 4KB | 서버 허용 한도 |
| 속도 | 빠름 | 느림 (메모리 조회) |
| 보안 | 취약 | 쿠키보다 높음 |

## 단점 요약

- 세션 ID 탈취 위험 (IP 바인딩으로 일부 방어, 한계 존재)
- **서버 부하**: 요청마다 세션 저장소 조회
- **수평 확장 어려움**: 세션 공유 or 중앙 세션 저장소(Redis) 필요
