# RAG vs LLM Wiki Pattern — 트레이드오프 비교

> AI에게 외부 지식을 주는 두 가지 패턴. 우열이 아니라 **자료의 성격과 목적**에 따라 선택한다.

## 두 패턴의 본질적 차이

### [[rag]] — "검색해서 가져오기"

```
[원본 문서] → chunk화 → 벡터 DB
                          ↓ (질문 시)
[질문] → 벡터 검색 → top-k chunk → LLM → 답변
```

**원본이 곧 데이터.** 매 질문마다 원본을 재검색한다.

### [[llm-wiki-pattern]] — "컴파일해서 누적하기"

```
[원본 문서] → AI가 읽고 컴파일 → [위키 페이지들 + 연결]
                                       ↓ (질문 시)
[질문] → 위키 index → 관련 페이지 → LLM → 답변 → (가치 있으면) 새 페이지로 환류
```

**위키가 데이터.** 위키는 원본을 읽고 만들어진 **2차 자산**이며, 질문도 위키를 키운다.

## 비교표

| 항목 | [[rag]] | [[llm-wiki-pattern]] |
| --- | --- | --- |
| 자료 형태 | chunk (단편) | 페이지 + wikilink (구조) |
| 자료 생성자 | 자동(chunking) | AI가 컴파일 |
| 갱신 빈도 | 원본 추가 시 자동 인덱싱 | AI가 ingest 작업 수행 |
| 질문 처리 | 매번 벡터 검색 | index → 페이지 직접 읽기 |
| 출처 추적 | chunk 메타데이터 | wikilink·frontmatter |
| 연결성 | chunk 간 끊김 | 명시적 링크로 보존 |
| 누적성 | ❌ 질문은 휘발 | ✅ 질문도 페이지로 환류 |
| 자료 규모 한계 | 거의 무제한 | 컴파일 가능 범위 |
| 자동화 정도 | 높음 | 낮음 (AI의 작업 필요) |
| 추론 비용 | 매 질문마다 검색 | 위키 한 번 컴파일하면 가벼움 |

## 선택 기준

### RAG가 맞는 경우

- **자료가 방대하다** — 사내 위키 수만 페이지, 전체 코드베이스, 법령 전문
- **자료가 자주 바뀐다** — 매일 갱신되는 운영 문서·이슈 트래커
- **chunk 단위 답이면 충분** — "이 약관 몇 조에 X가 있나" 같은 위치 검색
- **운영 자동화 필요** — 사람이 일일이 정제할 수 없음

### LLM Wiki Pattern이 맞는 경우

- **자료가 정제 가능한 규모** — 개인 학습·연구 노트, 한 프로젝트의 핵심 개념
- **개념 간 연결이 가치** — "X는 왜 등장했나 / Y와 어떻게 다른가" 같은 관계 질문
- **누적이 목적** — 같은 질문을 반복하지 않고, 지식을 키우고 싶음
- **품질·신뢰가 중요** — chunk 노이즈를 감당할 수 없는 도메인

## 우리(NoteWiki)는 왜 Wiki 패턴인가

박영균의 비전:
> "자주 쓰는 개념과 그 **연결 관계**를 기록해 반복 학습하고, 맥락에 맞춰 꺼내주는 AI 색인 도구."

이 비전에서 진짜 가치는 **chunk가 아니라 연결**이다:

- "Tomcat → Servlet → Spring MVC가 어떻게 이어지나" — chunk 검색으론 답하기 어렵다.
- "Filter와 Interceptor의 차이" — 두 chunk를 합쳐도 비교가 되지 않는다.
- "내가 1년 전 공부한 X를 지금 실무 Y에 어떻게 적용하나" — 누적된 위키만이 답할 수 있다.

→ 그래서 NoteWiki는 [[llm-wiki-pattern]]을 따른다.

## 두 패턴은 합칠 수 있다

상호 배타가 아니다. 실제로 자료가 커지면 **하이브리드**가 자연스럽다:

```
[위키 (정제된 핵심 개념·연결)]  ← Wiki Pattern
[원본 raw/ 폴더 전체 (방대)]    ← RAG로 보완 검색
```

질문이 들어오면:
1. 먼저 위키 index를 본다 (빠르고 구조적)
2. 위키에 없으면 raw/에 RAG 검색
3. 검색 결과가 가치 있으면 위키 페이지로 환류

→ 이게 장기적으로 NoteWiki가 도달할 수 있는 모습이다. 현재는 자료가 작아 Wiki만으로 충분.
