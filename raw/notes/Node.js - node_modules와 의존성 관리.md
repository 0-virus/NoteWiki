맨 마지막을 보면 이런 파일이 있다.

package.json이 프로젝트에 사용되는 패키지 목록을 나열해 둔 파일임은 알고 있었는데, 이건 또 무슨 파일인가? 그리고 분명 루트 디렉토리에도 package-lock.json이 있었는데, 둘이 중복 파일인 건가? 이런 생각에 다시 살펴봤는데, node_modules 폴더 안의 파일에는 앞에 점이 붙어있다는 것을 확인했다. 다른 파일인 것 같기는 한데, 뭐가 다른 건지 몰라 지피티에게 물어봤다.

## 사전지식: node_modules

일단, 현재 프로젝트 파일의 node_modules는 프로젝트 전용으로 설치된 의존성의 목록이다. Node.js가 설치된 경로에도 node_modules 폴더가 있는데, 이는 전역 시스템(CLI 도구 등)에서 사용되는 의존성으로, 프로젝트 파일에 존재하는 폴더와는 별개이다. 마치 파이썬에서 가상 환경을 생성해 패키지 버전을 관리하듯이 Node.js도 폴더를 활용해 프로젝트별로 버전을 관리하는 것이다.

## 1. package.json

루트 디렉토리에 있는 해당 파일은 프로젝트의 메타데이터와 필요한 패키지 목록을 정의한다.

- 파일 예시
  ```json
  {
    "name": "blog",
    "private": true,
    "version": "0.0.0",
    "type": "module",
    "scripts": {
      "dev": "vite",
      "build": "vite build",
      "lint": "eslint .",
      "preview": "vite preview"
    },
    "dependencies": {
      "react": "^19.2.0",
      "react-dom": "^19.2.0"
    },
    "devDependencies": {
      "@eslint/js": "^9.39.1",
      "@types/react": "^19.2.5",
      "@types/react-dom": "^19.2.3",
      "@vitejs/plugin-react": "^5.1.1",
      "eslint": "^9.39.1",
      "eslint-plugin-react-hooks": "^7.0.1",
      "eslint-plugin-react-refresh": "^0.4.24",
      "globals": "^16.5.0",
      "prettier": "3.7.4",
      "vite": "^7.2.4"
    }
  }
  ```

기본적으로 위와 같고, 특징은 다음과 같다.

- 버전 범위 지정: ^, ~와 같은 범위로 버전을 지정할 수 있음
  ```json
  "react": "^18.3.1"   // 18.3.1 이상 ~ 19.0.0 미만
  "react": "~18.3.1"   // 18.3.1 이상 ~ 18.4.0 미만
  "react": "18.3.1"    // 정확히 18.3.1만
  "react": "*"         // 아무 버전이나
  ```
- Git에 반드시 포함되는 파일임
- 직접 설치한 패키지만 간결하게 나열되어 있음
- 직접 편집이 가능함(패키지 설치 시 자동으로 수정되기는 함)

## 2. package-lock.json

마찬가지로 루트 디렉토리에 저장되어 있는 해당 파일은 의존성 잠금 파일로, 설치된 모든 패키지의 **정확한 버전**을 기록한다.

- 파일 예시
  ```json
  {
    "name": "blog",
    "version": "0.0.0",
    "lockfileVersion": 3,
    "requires": true,
    "packages": {
      "": {
        "name": "blog",
        "version": "0.0.0",
        "dependencies": {
          "react": "^19.2.0",
          "react-dom": "^19.2.0"
        },
        "devDependencies": {
          "@eslint/js": "^9.39.1",
          "@types/react": "^19.2.5",
          "@types/react-dom": "^19.2.3",
          "@vitejs/plugin-react": "^5.1.1",
          "eslint": "^9.39.1",
          "eslint-plugin-react-hooks": "^7.0.1",
          "eslint-plugin-react-refresh": "^0.4.24",
          "globals": "^16.5.0",
          "prettier": "3.7.4",
          "vite": "^7.2.4"
        }
      },
      "node_modules/@babel/code-frame": {
        "version": "7.27.1",
        "resolved": "https://registry.npmjs.org/@babel/code-frame/-/code-frame-7.27.1.tgz",
        "integrity": "sha512-cjQ7ZlQ0Mv3b47hABuTevyTuYN4i+loJKGeV9flcCgIK37cCXRh+L1bd3iBHlynerhQ7BhCkn2BPbQUL+rGqFg==",
        "dev": true,
        "license": "MIT",
        "dependencies": {
          "@babel/helper-validator-identifier": "^7.27.1",
          "js-tokens": "^4.0.0",
          "picocolors": "^1.1.1"
        },
        "engines": {
          "node": ">=6.9.0"
        }
      }
      // ... 수백 개의 패키지 정보
    }
  }
  ```

핵심 특징은 다음과 같다.

- npm이 자동 생성 및 관리: 직접 수정하지 않음
- 범위가 아닌 정확한 버전이 표시됨
- 전체 의존성 트리: 직접 설치한 패키지뿐만 아니라 모든 하위 의존성을 포함
- Git에 반드시 포함되어 있음

이 파일이 필요한 이유에 대해 지피티는 다음 시나리오를 제시했다.

### package.json만 있다면?

```bash
# 개발자 A (2024년 1월)
npm install
# react@18.3.1 설치됨

# 개발자 B (2024년 6월) - 같은 package.json 사용
npm install
# react@18.5.0 설치됨 (^18.3.1 범위 내 최신 버전)

# 결과: 다른 버전으로 버그 발생 가능!
```

### package-lock.json이 있으면

```bash
# 개발자 A, B 모두
npm install
# 정확히 같은 버전 (18.3.1) 설치됨!
```

## 3. node_modules/.package-lock.json

해당 파일은 실제 설치된 node_modules의 상태를 캐싱하는 파일이다.

- 파일 예시
  ```json
  {
    "name": "blog",
    "version": "0.0.0",
    "lockfileVersion": 3,
    "requires": true,
    "packages": {
      "node_modules/@babel/code-frame": {
        "version": "7.27.1",
        "resolved": "https://registry.npmjs.org/@babel/code-frame/-/code-frame-7.27.1.tgz",
        "integrity": "sha512-cjQ7ZlQ0Mv3b47hABuTevyTuYN4i+loJKGeV9flcCgIK37cCXRh+L1bd3iBHlynerhQ7BhCkn2BPbQUL+rGqFg==",
        "dev": true,
        "license": "MIT",
        "dependencies": {
          "@babel/helper-validator-identifier": "^7.27.1",
          "js-tokens": "^4.0.0",
          "picocolors": "^1.1.1"
        },
        "engines": {
          "node": ">=6.9.0"
        }
      },
      "node_modules/@babel/compat-data": {
        "version": "7.28.5",
        "resolved": "https://registry.npmjs.org/@babel/compat-data/-/compat-data-7.28.5.tgz",
        "integrity": "sha512-6uFXyCayocRbqhZOB+6XcuZbkMNimwfVGFji8CTZnCzOHVGvDqzvitu1re2AU5LROliz7eQPhB8CpAMvnx9EjA==",
        "dev": true,
        "license": "MIT",
        "engines": {
          "node": ">=6.9.0"
        }
      }
      // package-lock.json과 거의 동일한 내용
    }
  }
  ```

특징은 다음과 같다.

- npm이 자동 생성: npm 5.7.0 이후
- .으로 시작하는 파일 → 숨김 파일임
- Git 무시: .gitignore에 자동 포함. 위의 폴더 트리([여기](https://www.notion.so/Node-js-node_modules-2e0b563074a6801aac57d7f489af248e?pvs=21))를 보면 node_modules 폴더 자체가 회색으로 뜨는 것을 보아 Vite이 프로젝트를 빌드할 때 자동으로 무시하도록 설정하는 듯함

해당 파일이 필요한 이유는 다음과 같다.

### npm install 실행 시 프로세스:

```bash
npm install

# 1단계: node_modules/.package-lock.json 확인
if (node_modules/.package-lock.json과 package-lock.json이 일치) {
    // ✅ 이미 올바르게 설치됨 → 스킵!
    console.log("up to date");
} else {
    // ❌ 불일치 → 재설치 필요
    // 2단계: package-lock.json 기반으로 패키지 다운로드
    // 3단계: node_modules/.package-lock.json 업데이트
}
```

### 성능 비교:

```bash
# 첫 설치 (node_modules 없음)
npm install  # 30초 소요

# 두 번째 실행 (.package-lock.json가 있으면)
npm install  # 1초 소요 ("up to date" 메시지만)
```

## 전체 프로세스 시나리오

### 시나리오 A: 개발자 A (최초 개발)

1. 프로젝트 생성

   ```bash
   npm create vite@latest blog
   ```

2. 패키지 자동 설치됨

   ```
   ✅ package.json (자동 생성)
   ✅ package-lock.json (자동 생성)
   ✅ node_modules/ (자동 생성)
   ✅ node_modules/.package-lock.json (자동 생성)
   ```

3. 추가 패키지 설치

   ```bash
   npm install axios
   ```

   패키지 설치 후 변화:

   ```
   ✅ package.json에 axios 추가 (자동)
   ✅ package-lock.json 업데이트 (자동)
   ✅ node_modules/axios/ 생성 (자동)
   ✅ .package-lock.json 업데이트 (자동)
   ```

4. Git 커밋

   ```bash
   git add .
   git commit -m "Initial commit"
   ```

   Git에 포함되는 것:

   ```
   ✅ package.json
   ✅ package-lock.json
   ✅ src/
   ❌ node_modules/ (gitignore에 의해 제외)
   ```

### 시나리오 B: 개발자 B (프로젝트 복제)

1. 프로젝트 복제

   ```bash
   git clone https://github.com/project/blog.git
   cd blog
   ```

   현재 상태:

   ```
   ✅ package.json (Git에서 받음)
   ✅ package-lock.json (Git에서 받음)
   ✅ src/ (Git에서 받음)
   ❌ node_modules/ (없음!)
   ```

2. 의존성 설치

   ```bash
   npm install
   ```

   npm이 하는 일:
   1. package-lock.json 읽기
      └> "react@18.3.1, axios@1.6.0 설치 필요"
   2. 네트워크에서 다운로드
      └> npm 레지스트리에서 패키지 가져오기
   3. node_modules/ 생성 및 압축 해제
      └> node_modules/react/
      └> node_modules/axios/
      └> node_modules/... (수백 개)
   4. .package-lock.json 생성
      └> 설치 완료 상태 기록

**결과: 개발자 A와 100% 동일한 환경!**

### 시나리오 C: 개발자 A (추가 개발 후)

1. 6개월 후, 패키지 추가

   ```bash
   npm install redux
   ```

   패키지 추가 후 변화:

   ```
   ✅ package.json: redux 추가 (자동)
   ✅ package-lock.json: redux 정보 추가 (자동)
   ✅ node_modules/redux/ 추가 (자동)
   ✅ .package-lock.json 업데이트 (자동)
   ```

2. Git 푸시

   ```bash
   git add package.json package-lock.json
   git commit -m "Add redux"
   git push
   ```

### 시나리오 D: 개발자 B (변경사항 반영)

1. 최신 변경사항 받기

   ```bash
   git pull
   ```

   받은 것:

   ```
   ✅ package.json (redux 추가됨)
   ✅ package-lock.json (redux 정보 있음)
   ❌ node_modules/ (Git에 없음, 여전히 예전 상태)
   ```

2. 의존성 동기화

   ```bash
   npm install
   ```

   npm이 하는 일:
   1. .package-lock.json vs package-lock.json 비교
      └> "redux가 없네? 추가 필요!"
   2. redux만 다운로드 (기존 패키지는 그대로)
      └> node_modules/redux/ 추가
   3. .package-lock.json 업데이트
      └> redux 추가된 상태로 기록

**결과: 개발자 A와 다시 100% 동일!**

# Appendix: 유용한 패키지 설치 명령어

```bash
# 일반 의존성 (배포 시 필요)
npm install axios
npm i axios             # 축약형

# 개발 의존성 (개발 시에만 필요)
npm install -D eslint
npm install --save-dev eslint

# 특정 버전 설치
npm install react@18.3.1

# 패키지 제거
npm uninstall axios     # package.json도 자동 업데이트!

# 전역 설치 (CLI 도구)
npm install -g create-vite

### 의존성 동기화

# 보통은 이것만 사용
npm install

# CI/CD에서는 (더 빠르고 정확)
npm ci
# → package-lock.json 기반으로만 설치
# → node_modules 완전 삭제 후 재설치
# → package.json 수정 안 함
```
