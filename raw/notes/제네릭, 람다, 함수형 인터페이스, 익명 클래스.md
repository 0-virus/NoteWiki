# 1. 제네릭(Generics) 정리

## 1-1. 제네릭이란?

- **클래스/인터페이스/메서드가 사용할 타입을 외부에서 파라미터처럼 받는 기능**
- 타입을 변수처럼 다루는 느낌 → `T`, `E`, `K`, `V` 같은 타입 매개변수 사용

```java
Copyclass Box<T> {
    private T value;
    public void set(T value) { this.value = value; }
    public T get() { return value; }
}

Box<String> sBox = new Box<>();
Box<Integer> iBox = new Box<>();
```

- `Box<String>` → `T` 가 `String` 으로 치환된 버전
- `Box<Integer>` → `T` 가 `Integer` 로 치환된 버전

## 1-2. 제네릭을 쓰는 이유

1. **컴파일 타임 타입 체크**
   - 잘못된 타입 사용을 컴파일 시점에 잡아줌
   - 런타임 `ClassCastException` 줄어듦
2. **캐스팅(형변환) 없이 사용 가능**

   ```java
   CopyList list = new ArrayList();       // 옛날 방식
   Object o = list.get(0);            // 꺼낼 때마다 형변환 필요

   List<String> list2 = new ArrayList<>();
   String s = list2.get(0);           // 형변환 불필요
   ```

3. **타입 안정성 + 가독성 향상**

## 1-3. 제네릭의 기본 규칙

- **참조 타입만 가능**: `int` X → `Integer` O
- 타입 매개변수 관습적 네이밍:
  - `T` : Type
  - `E` : Element (List 등)
  - `K`, `V` : Key, Value(Map)
  - `N` : Number
- 여러 개도 가능: `<T, U>`, `<K, V>`

# 2. 함수형 인터페이스 & 람다

## 2-1. 함수형 인터페이스

- **추상 메서드가 딱 하나인 인터페이스**
- “함수 하나”를 표현하기 위한 틀
- 람다식의 **타겟 타입**이 됨

```java
Copy@FunctionalInterface
interface IAdd<T> {
    T add(T x, T y);
}
```

- 여기서 `IAdd<T>` 는:
  - “T 두 개 받아서 T 하나 반환하는 함수” 란 의미의 타입

## 2-2. 제네릭 + 함수형 인터페이스

```java
CopyIAdd<Integer> intAdd = (x, y) -> x + y;
IAdd<String> strAdd = (a, b) -> a + b;

int r1 = intAdd.add(10, 20);          // 30
String r2 = strAdd.add("a", "b");     // "ab"
```

- `IAdd<Integer>` → `Integer add(Integer, Integer)`
- `IAdd<String>` → `String add(String, String)`
- **제네릭이 람다의 매개변수/리턴 타입을 결정**해줌

## 2-3. 람다식이 하는 일 (기본 개념)

**람다식:**

```java
CopyIAdd<Integer> o = (x, y) -> x + y;
```

**짧은 버전:**

```java
CopyIAdd<Integer> o = new IAdd<Integer>() {
    @Override
    public Integer add(Integer x, Integer y) {
        return x + y;
    }
};
```

즉,

- 람다 = **함수형 인터페이스의 “유일한 메서드 구현”을 간단히 표현한 문법**
- 실제로는 **객체 하나가 만들어진다** (익명 클래스랑 비슷한 개념)

---

# 3. 람다 vs 익명 클래스 정리

## 3-1. 공통점

- 둘 다 **인터페이스 구현체를 즉석에서 만들어서 변수에 담는다**
- 아래 두 코드는 “행동” 면에서 거의 동일

```java
Copy// 익명 클래스
IAdd<Integer> a = new IAdd<Integer>() {
    @Override
    public Integer add(Integer x, Integer y) {
        return x + y;
    }
};

// 람다
IAdd<Integer> b = (x, y) -> x + y;
```

---

## 3-2. 중요한 차이들

### (1) `this` 가 가리키는 대상

**익명 클래스**

```java
Copyclass Outer {
    int v = 1;

    void test() {
        IAdd<Integer> a = new IAdd<Integer>() {
            int v = 2;

            @Override
            public Integer add(Integer x, Integer y) {
                System.out.println(v);        // 2 (익명 클래스의 v)
                System.out.println(this.v);   // 2 (익명 클래스의 v)
                System.out.println(Outer.this.v); // 1 (외부 Outer의 v)
                return x + y;
            }
        };
    }
}
```

- `this` → **익명 클래스 인스턴스**
- 외부 클래스는 `Outer.this` 로 접근

**람다**

```java
Copyclass Outer {
    int v = 1;

    void test() {
        IAdd<Integer> b = (x, y) -> {
            int v = 2;
            System.out.println(v);      // 2 (람다 안의 지역 변수)
            System.out.println(this.v); // 1 (Outer의 v)
            return x + y;
        };
    }
}
```

- 람다는 **새로운 this를 만들지 않음**
- `this` → **바깥 클래스 인스턴스(Outer)**

→ 정리:

- 익명 클래스: `this` = “새로 생긴 클래스”
- 람다: `this` = “바깥 객체”

---

### (2) 스코프(범위) & 변수 선언

- **익명 클래스**는 진짜 클래스를 하나 더 만드는 거라서:
  - 안에 **필드**를 선언할 수 있음 (`int v = 2;` → 필드)
- **람다**는 메서드 안의 코드 블록 같은 취급이라서:
  - 안에서 선언하는 건 **지역 변수**뿐

예)

```java
Copyvoid test() {
    int v = 10;

    // 람다
    IAdd<Integer> b = (x, y) -> {
        // int v = 20; // ❌ 에러: 같은 메서드 스코프에 v가 이미 있음
        return x + y + v;
    };
}
```

- 람다는 바깥 메서드(`test`)의 지역 변수 스코프를 공유해서같은 이름의 지역 변수 다시 선언 불가.

반면 익명 클래스는 클래스 스코프가 따로라서 필드 선언 가능:

```java
Copyvoid test() {
    int v = 10;

    IAdd<Integer> a = new IAdd<Integer>() {
        int v = 20; // 이건 익명 클래스의 필드

        @Override
        public Integer add(Integer x, Integer y) {
            return x + y + v; // 20 사용
        }
    };
}
```

---

### (3) 어디에 쓸 수 있나

- **익명 클래스**
  - 메서드 여러 개 있는 인터페이스도 구현 가능
  - 추상 클래스 상속도 가능
- **람다**
  - **추상 메서드 1개짜리 인터페이스(함수형 인터페이스)** 에만 사용 가능

---

### (4) 내부적으로

- 익명 클래스
  - 컴파일 시 **새 클래스(내부 클래스) 파일**이 생성되는 느낌
- 람다
  - JVM이 더 최적화된 방식(`invokedynamic` 등)으로 처리
  - 문법적으로는 함수 표현에 특화

※ 실무에서 보통은 “문법 + 사용제약 + this 차이” 정도만 알아도 충분.

# 4. 제네릭 + 람다 + 함수형 인터페이스 조합 예제 요약

예제 코드:

```java
Copy// 제네릭으로 타입을 받아, 해당 타입의 두 값을 더하는 인터페이스
interface IAdd<T> {
    public T add(T x, T y);
}

public class Main {
    public static void main(String[] args) {
        // 제네릭을 통해 람다 함수의 타입을 결정
        IAdd<Integer> o = (x, y) -> x + y;

        int result = o.add(10, 20);
        System.out.println(result); // 30
    }
}
```

**해석 정리:**

1. `IAdd<T>`

   → “T 두 개 받아서 T 하나 돌려주는 함수”라는 의미의 함수형 인터페이스

2. `IAdd<Integer> o`

   → `T` = `Integer`

   → `Integer add(Integer x, Integer y)` 라는 함수 규격으로 확정

3. `(x, y) -> x + y`

   → 이 규격에 맞는 **함수 구현**을 람다식으로 표현

   → 내부적으로는 `IAdd<Integer>` 구현 객체 하나 만들어짐

4. `o.add(10, 20)`
   - `10`, `20` 은 `int` → `Integer` 로 오토박싱
   - `x + y` 할 때는 `int` 덧셈 (언박싱 후 연산)
   - 결과 `Integer` 로 리턴 → 다시 `int result` 에 언박싱

---

# 5. 한 줄 요약

- **제네릭**
  → 타입을 파라미터처럼 받아서, 안전하고 재사용 가능한 코드 작성
- **함수형 인터페이스**
  → “함수 하나”를 표현하기 위한 틀 (추상 메서드 1개)
- **람다식**
  → 함수형 인터페이스의 **유일한 메서드 구현을 짧게 쓰는 문법**
  → 제네릭과 함께 쓰면, **람다의 매개변수/리턴 타입이 제네릭 타입으로 자동 결정**
- **람다 vs 익명 클래스**
  - 공통: 둘 다 “즉석에서 구현체 객체 하나 만든다”
  - 차이 핵심:
    - `this`: 익명 클래스 → 자기 자신 / 람다 → 외부 객체
    - 람다는 함수형 인터페이스에만 사용 가능
    - 스코프/필드 선언 방식이 다름 (람다는 지역 변수 스코프 공유)
