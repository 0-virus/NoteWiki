# 도커란?

컨테이너 수준의 가상화 방식으로 소프트웨어를 배포하는 방식의 PaaS

## 컨테이너

소프트웨어 개발에 필요한 코드, 라이브러리, 환경설정 파일등을 모아 패키지화한 것.

## 가상화?

컴퓨터 리소스를 추상화한다?

### 가상화 종류

1. 호스트 가상화: 호스트 운영체제를 두고 하이퍼바이저 위에 여러 가상 머신을 띄워 각각 게스트 운영체제를 둠.
2. 하이퍼바이저 가상화: 호스트 운영체제 없이 가상 머신을 띄움. 성능이 우수하지만 초기 설정과 관리가 어려움.
3. 컨테이터 가상화: 운영체제 위에 도커 설치 후 컨테이너를 통해 앱을 실행함.

## 도커의 구성 요소

- docker-cli: 도커 클라이언트. 터미널 같은 존재.
- dockerd: docker daemon. 백그라운드에서 실행되는 데몬 프로세스. 도커 api 요청 수신 및 도커 객체 관리.
- containerd: 컨테이너 실행 및 관리를 수행하는 고수준 런타임. 컨테이너의 생명 주기 전체를 다룸.
- runc: 컨테이너 실행만 담당하는 저수준 컨테이너 런타임.
- containerd-shim: containerd와 runc 사이에서 작동하는 중간 프로세스.

## 도커 작동 방식

- 도커 클라이언트: 도커에 명령을 내릴 수 있는 CLI 도구.
- 도커 호스트: 도커를 설치한 서버/가상머신
- 도커 레지스트리: 도커 이미지를 저장/배포하는 시스템. 크게 공개/개인 레지스트리로 나뉨. 도커 허브가 대표적인 공개 레지스트리.

### 도커 이미지

- 컨테이너 형태로 소프트웨어를 배포하기 위해 필요한 모든 요소를 실행할 수 있는 포맷으로 컴파일 및 빌드한 **패키지**
- 독립적이므로 의존성을 고려할 필요가 없음
- 특정 시점의 도커 컨테이너 상태를 담은 스냅샷과 같음

### 도커 컨테이너

- 도커 이미지를 실행할 수 있는 인스턴스
- 자체적으로 파일 시스템을 가지고 있음
- 각 컨테이너는 독립적으로 실행됨
- 각 컨테이너는 가상 머신처럼 운영체제를 전부 포함하는 게 아니라, 도커 엔진과 운영체제를 공유함 -> 컨테이너는 생각보다 가벼움!

### `docker run hello-world` 치면 무슨 일이?

1. `docker run`: 컨테이너 실행
2. 로컬에서 hello-world:latest를 찾음 -> 없음
3. library/hello-world에서 pull을 받음

- digest: 도커 이미지의 식별값(해시화된 값)

## 도커 이미지 기초 명령어

- `docker image pull [이미지 이름]:[태그 이름]` / `docker image pull [이미지 이름]@[DIGEST 값]` : 도커 이미지 다운로드하기. 태그 이름 지정하지 않으면 자동으로 latest로 다운로드함
- `docker image ls` : 도커 이미지 목록 확인

## 도커 이미지 구조

- 이미지 인덱스: 매니페스트 리스트라고 보면 됨. pull 시 나오는 digest가 얘꺼임.
- 이미지 매니페스트: 이미지 실행을 위한 설정 파일과 여러 레이어들로 구성됨. 이미지마다 다수의 매니페스트로 이루어짐.
- 레이어: 레이어들은 각각의 digest 값을 가짐.

## 도커 컨테이너 실행

- `docker container run [이미지명]` : 도커 호스트의 도커 데몬이 호출되어 도커 호스트에 있는 이미지를 컨테이너 형태로 실행함.
- `docker container ls` : 현재 실행 중인 컨테이너 목록 확인.
- `docker container ls -a` : 종료된 컨테이너까지 모두 확인.
  - 컨테이너 실행 시, 컨테이너 내부 프로세스가 모두 종료되면 해당 컨테이너는 중료됨. 즉, `docker container ls`에 안 잡힘!
