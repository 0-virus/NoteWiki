# 인증 방식

## Cookie

쿠키: Key - Value 문자열 덩어리
클라이언트가 웹사이트 방문 시, 클라이언트의 브라우저에 설치되는 기록 파일

### Cookie 인증 방식

1. 클라이언트가 서버에 요청을 보냄
2. 서버는 응답 작성 시 클라이언트 측에 저장하고 싶은 정보를 응답 헤더의 Set-Cookie에 담아 보냄
3. 클라이언트는 서버에 다시 요청 시 헤더에 쿠키를 담아 보냄

### 단점

- 보안에 취약함(쿠키 조작 및 유출)
  - 쿠키는 브라우저 내부에 저장됨
    -> document.cookie로 쿠키에 접근이 가능
    -> 계정 접근 권한 등을 탈취할 위험이 있음
- 쿠키는 용량 제한이 있음
- 브라우저마다 쿠키 지원 형태가 다름
- 쿠키 사이즈가 클 수록 네트워크 부하가 커짐

### Secure 쿠키 vs HttpOnly 쿠키

- HttpOnly: 자바스크립트로 쿠키에 접근하지 못하도록 막는 옵션
- Secure: HTTPS에서만 쿠키를 전송할 수 있음

## Session 인증

클라이언트의 민감한 정보를 서버 측에 저장하고 관리함!

### 인증 방식

1. 유저가 로그인 -> 세션이 서버 메모리 or DB에 저장됨(세션 id 기준으로 정보를 저장)
2. 서버는 브라우저 쿠키에 Session ID를 저장함
3. 브라우저(클라이언트)는 이후 요청에 Session ID를 쿠키에 담아 함께 보냄
4. 서버는 클라이언트가 보낸 Session ID와 서버에서 관리하는 Session ID를 비교하여 인증을 수행함

### 단점

- 세션 ID 탈취당할 위험이 있음
  - 서버가 클라이언트 IP를 같이 저장함으로써 해결 가능
    <한계>
    공격자가 같은 NAT/VPN 안에 있으면 IP가 같아 보일 수 있음
    - 사용자가 모바일 -> WIFI 전환 시, IP가 바뀌어 로그아웃되는 문제가 발생함
    - 기업 내부망은 여러 사용자가 같은 출구 IP를 공유함
- 트래픽이 심할 경우 서버 부하가 커짐

## Cookie vs Session

|           | Cookie                              | Session                   |
| --------- | ----------------------------------- | ------------------------- |
| 저장 위치 | 클라이언트                          | 서버                      |
| 저장 형식 | 텍스트                              | Object                    |
| 만료 시점 | 쿠키 저장 시 설정                   | 브라우저 종료 시 삭제     |
| 용량 제한 | 총 300개, 도메인당 20개, 쿠키당 4KB | 서버가 허용하는 한 무제한 |
| 속도      | 세션보다 빠름                       | 느림                      |
| 보안      | 취약                                | 쿠키보다 높음             |

## Token 인증

클라이언트가 서버 접속하면, 서버에서 토큰을 부여함. 이 토큰을 요청 시마다 헤더에 심어서 보내면 서버는 발급한 토큰과의 일치 여부를 판별함.

### 장점

- 세션 인증은 서버가 세션 정보를 가지고 있고 이를 조회해야 하기 때문에 오버헤드가 많이 발생함. But 토큰 방식은 클라이언트가 정보를 저장하고 있기 때문에 서버는 위조 여부만 판별하면 됨.
- Stateless함. (서버가 클라이언트의 상태를 기억하지 않는다는 뜻)

### 단점

- 토큰 자체의 길이가 길기 때문에, 요청이 많을 수록 네트워크 부하가 심해질 수 있음.
- Payload는 암호화되지 않으므로 클라이언트의 민감한 정보를 담을 수 없음.
- 토큰 탈취 시 대처가 어려움

## JWT

- 인증에 필요한 정보들을 암호화시킨 JSON 토큰을 의미함.
- JSON 정보를 Base64 URL-safe Encode를 사용해 직렬화함.
  - Base64 URL-safe Encode? -> 일반적 Base64 Encode에서 '+', '/'를 각각 '-', '\_'로 표현해 URL에서 오류 없이 사용하도록 만든 것

### JWT 구조

- raw/assets/jwt.png 참고
- Header, Payload, Signature로 구성
- Header: 타입과 해시 알고리즘 종류
- Payload: 서버가 첨부한 사용자 권한 정보 및 데이터
- Signature: Header와 Payload를 인코딩한 후 해시 함수를 적용시키고 개인키로 서명한 전자서명

### JWT 인증 과정

1. 클라이언트가 로그인 요청
2. 서버가 요청 받으면 세 가지 정보 정의 후 암호화. 쿠키에 담아서 클라이언트에게 발급.
3. 클라이언트는 토큰을 local storage에 저장. 이후 요청의 인증 헤더에 엑세스 토큰을 담아서 전송.
4. 서버는 클라이언트가 전송한 토큰이 서버가 발급한 토큰과 일치하는지 판별. 문제 없으면 응답
5. 엑세스 토큰이 만료되면 클라이언트는 리프레시 토큰을 이용해 새로운 엑세스 토큰을 발급받음.

### 참고

- JWT는 정보 보호가 목적이 아니라 서명/위조 방지가 목적임.
- Base64로 인코딩하기 때문에 Payload의 데이터는 복호화를 통해 쉽게 드러남.
- 따라서 Payload에 비밀번호 등 민감한 정보를 담으면 안됨.

### 토큰이 신뢰성을 가지는 이유

- A(Header) + B(Payload) + C(Signature) 라고 했을 때
- 다른 유저가 B를 B'로 임의로 조작하면,
- A + B' + C 보냄
- -> 서버가 유효성 검증 해보면 A + B' + C'가 됨
- -> 서명이 맞지 않음!

### 장점

- 인증을 위한 별도 저장소 필요 없음
- Stateless이므로 서버 확장성 우수
- 모바일에서도 잘 동작함(모바일은 세션 사용 불가)

### 단점

- Self-contained: 토큰 자체가 정보를 담고 있음
- 토큰 길이 증가 -> 네트워크 부하
- 토큰 탈취 위험

### Refresh Token

- 엑세스 토큰에 짧은 수명을 주는 대신, 계속 로그인 안 해도 되도록 엑세스 토큰 만료 시 재발급해주기 위한 토큰
- 로그인 성공 시 서버가 Access Token / Refresh Token을 동시에 발급
- 클라이언트가 만료된 Access Token을 보내면 서버는 같이 보내 준 Refresh Token을 DB에 저장된 것과 비교해 일치할 경우 Access Token을 다시 발급
