`## 💭 내 메모`의 3가지 질문에 답변.

**Q1. ALTER TABLE 락 대기 구조**

- 열린 트랜잭션 없을 때: ALTER TABLE → 즉시 Exclusive MDL 획득 → 실행 → 완료. 뒤 쿼리는 DDL 실행 시간만큼만 대기.
- 열린 트랜잭션 있을 때: 트랜잭션이 Shared MDL 보유 → ALTER TABLE이 Exclusive MDL 획득 불가 → WAITING → 뒤에 오는 모든 쿼리도 대기열에 쌓임.

**Q2. 용어 정리**

- 메타데이터 락(MDL): 테이블 구조 보호 락. SELECT/DML = Shared MDL, DDL = Exclusive MDL
- 배타적 락: 다른 접근을 완전히 차단하는 락 (↔ 공유 락: 여럿이 동시 읽기 가능)
- 트리거: INSERT/UPDATE/DELETE 이벤트 시 자동 실행되는 SQL 블록
- 마스터-슬레이브 시차(Replication Lag): 복제 구조에서 마스터 변경이 슬레이브에 반영되는 지연

**Q3. pt-osc 원리**: 새 테이블 생성 → 트리거로 실시간 동기화 → 청크 단위 데이터 복사 → RENAME으로 교체 (락은 RENAME 순간만). 단점: 트리거로 쓰기 2배 → 복제 지연. FK/기존 트리거 있으면 추가 설정 필요.

**Claude**

**1. 맞음.** Shared MDL끼리는 공존 → 동시 실행. ALTER TABLE(Exclusive MDL)이 WAITING 상태에 진입하는 순간, 그 뒤의 Shared MDL 요청까지도 대기열에 쌓임.

**2. pt-osc 사용법**

```bash
pt-online-schema-change \
  --alter "ADD COLUMN last_login_at DATETIME NULL" \
  --host=localhost --user=root --password=secret \
  D=mydb,t=users \
  --execute
```

주요 옵션: `--chunk-size`(부하 조절), `--max-load`(부하 초과 시 일시 중지), `--no-drop-old-table`(안전망)

**3. 영상 결론**: 점검 모드가 아님. "운영 DB에서 DDL은 코드 변경이 아니라 배포다." → 트래픽 패턴·열린 트랜잭션·복제 구조까지 고려. 대용량 테이블은 pt-osc 또는 MySQL 8.0 Online DDL 사용.
