## 대화에서 정리된 지식

@GeneratedValue는 @Id 값을 JPA/DB가 자동 생성하게 합니다. 새 엔티티를 만들 때 id는 직접 넣지 않고 null 상태로
저장해야 합니다. GenerationType.IDENTITY를 쓰면 DB의 auto increment 값이 저장 후 엔티티에 반영됩니다. id에 아
무 값이나 직접 넣는 방식은 새 엔티티/기존 엔티티 판단을 꼬이게 만들 수 있습니다.

회원가입 구현에서는 AuthController가 @RestController, @RequestMapping("/api/v1/auth"), @PostMapping("/signup")
구조를 가져야 합니다. 요청 DTO에는 @Valid @RequestBody를 붙여야 RegisterRequest의 검증 어노테이션이 동작합니
다. 회원가입 서비스는 User 저장, Account 저장까지 해야 하며, 단순히 객체만 만들면 id가 null이고 DB에도 반영되
지 않습니다.

MySQL [28000][1045] Access denied for user 'root'@'localhost'는 application.yaml의 DB 계정/비밀번호와 실제
MySQL 계정 정보가 다를 때 발생합니다. 프로젝트 설정은 root / 11111111인데, 로컬 MySQL이 3306을 점유하고 있으면
Docker MySQL이 아니라 로컬 MySQL에 접속하게 됩니다. 3306 포트는 netstat -ano | Select-String ':3306'로 확인했
고, 당시 mysqld가 점유 중이었습니다.

Docker MySQL 실행 시 ports are not available 오류는 이미 3306 포트를 다른 프로세스가 잡고 있어서 발생합니다.
Windows에서는 Stop-Service mysqld, net stop mysqld, 필요 시 Set-Service mysqld -StartupType Manual로 로컬
MySQL 서비스를 멈출 수 있습니다. 이후 docker compose up -d를 다시 실행하면 됩니다.

Docker 컨테이너 이름 충돌 오류는 같은 이름의 virtuber-mysql 컨테이너가 이미 존재할 때 발생합니다. 기존 컨테이
너를 재사용하려면 docker start virtuber-mysql, 삭제 후 다시 만들려면 docker stop virtuber-mysql, docker rm
virtuber-mysql, docker compose up -d 순서로 처리합니다.

Swagger UI는 springdoc-openapi-starter-webmvc-ui 의존성을 추가하면 기본적으로 /swagger-ui/index.html에서 볼 수
있습니다. Spring Security를 사용하면 /swagger-ui/**, /v3/api-docs/**, /swagger-ui.html을 permitAll()에 추가해
야 합니다. JWT Authorize 버튼을 띄우려면 OpenAPI SecurityScheme에 bearer auth 설정을 추가해야 합니다.

localhost에 대한 액세스가 거부됨은 서버가 안 떠 있거나, Spring Security가 Swagger 경로를 막거나, 포트가 다를
때 발생할 수 있습니다. 현재 서버 포트는 8000으로 설정했으므로 Swagger 주소는
http://localhost:8000/swagger-ui/index.html입니다.

애플리케이션 실행 실패 로그에서 핵심 원인은 ddl-auto: updateo 오타였습니다. Hibernate가 허용하는 값은 update,
create, create-drop, validate, none 등이고 updateo는 인식하지 못합니다. 그래서 EntityManagerFactory 생성이 실
패하고, 그 여파로 UserRepository, JwtAuthenticationFilter, Tomcat까지 연쇄 실패했습니다.

Swagger UI가 빈 화면이었던 원인은 JwtAuthenticationFilter에서 filterChain.doFilter(request, response)를 호출하
지 않았기 때문입니다. 필터에서 다음 필터/컨트롤러/정적 리소스로 요청을 넘기지 않으면 Swagger HTML/JS가 정상 응
답되지 않습니다.

로그인 요청이 맞는 ID/PW로도 403이 뜬 원인은 컨트롤러 매핑 오타였습니다. SecurityConfig는 /api/v1/auth/signin
을 허용했는데, 컨트롤러에는 /siginin으로 등록되어 있었습니다. 이 경우 /signin은 컨트롤러가 없고, /siginin은
permitAll 대상이 아니어서 차단될 수 있습니다.

Swagger의 Authorize에는 access token을 넣습니다. bearer auth 설정을 했으면 보통 토큰 문자열만 넣으면 되고, UI
가 직접 요구하면 Bearer {token} 형식으로 넣습니다. Refresh token은 Swagger Authorize에 넣는 값이 아닙니다.

/api/v1/stocks가 토큰 없이는 되고 토큰을 넣으면 403이 뜬 원인은 JWT 필터의 사용자 조회 방식이 잘못됐기 때문입
니다. 토큰 subject에는 user.id가 들어가는데, 필터에서 findByUserId(subject)로 로그인 ID처럼 조회하고 있었습니
다. findById(Long.valueOf(subject))로 조회해야 합니다. 또한 공개 API는 shouldNotFilter()에서 JWT 필터 자체를
제외할 수 있습니다.

Refresh token은 DB에서 관리하는 것이 좋습니다. Access token은 짧게 유지하고 서버 저장 없이 검증하며, refresh
token은 DB에 저장해서 재발급과 로그아웃에 사용합니다. 로그인 시 access token과 refresh token을 함께 발급하고,
refresh token은 refresh_tokens 테이블에 저장합니다.

Refresh token은 로그인 상태 표시용이 아니라 access token 재발급용입니다. 프론트에서 로그인 여부는 “access
token이 유효한지”, 또는 “refresh token으로 재발급이 성공하는지”로 판단해야 합니다. Refresh token이 존재한다고
해서 반드시 로그인 상태라고 볼 수는 없습니다. DB에서 삭제됐거나 만료됐거나 서버 secret이 바뀌었을 수 있습니다.

로그아웃 API가 refresh token만 삭제하면 이미 발급된 access token은 만료 전까지 이론상 유효합니다. 따라서 프론
트는 로그아웃 성공 시 로컬 access token과 refresh token을 모두 삭제해야 합니다. 즉시 서버 무효화까지 하려면
access token blacklist나 token version 같은 추가 설계가 필요합니다. MVP에서는 access token 만료 시간을 짧게 두
고, 로그아웃 시 프론트 저장 토큰을 삭제하는 방식이 현실적입니다.

signout에서 500이 뜰 수 있는 주요 원인은 refreshTokenRepository.deleteByToken(...)이 트랜잭션 없이 실행되는 경
우입니다. 삭제 쿼리는 서비스 메서드에 @Transactional을 붙여야 합니다. 또한 /api/v1/auth/signout, /api/v1/auth/
reissue는 공개 API로 처리한다면 JWT 필터의 shouldNotFilter()에도 추가해야 합니다.

/api/v1/accounts/me는 클라이언트가 userId를 보내면 안 됩니다. UserRequest를 만들지 않고, JWT 필터가
Authentication에 넣은 principal에서 userId를 꺼내야 합니다. 현재 필터는
UsernamePasswordAuthenticationToken(user.getId(), null, List.of()) 형태로 principal에 userId를 넣습니다. 컨트
롤러에서는 Long userId = (Long) authentication.getPrincipal();로 꺼냅니다.

/api/v1/accounts/me는 permitAll() 목록에 없기 때문에 자동으로 JWT 필터를 거칩니다. 별도로 필터를 호출할 필요는
없습니다. 인증이 없으면 컨트롤러까지 오지 않고 Spring Security에서 차단됩니다.

계좌 조회 응답에는 AccountMeResponse와 HoldingResponse가 적절합니다. AccountMeResponse는 현금 잔고, 총 매입금
액, 주식 평가금액, 총 자산, 평가손익, 수익률, 보유 주식 목록을 포함합니다. HoldingResponse는 종목 ID, 종목 코
드, 종목명, 보유 수량, 평균 매입가, 현재가, 총 매입금액, 평가금액, 평가손익, 수익률을 포함합니다.

보유 종목별 계산식은 totalPurchaseAmount = averagePrice _ quantity, evaluationAmount = currentPrice _
quantity, profitAmount = evaluationAmount - totalPurchaseAmount, profitRate = totalPurchaseAmount == 0 ? 0 :
profitAmount / totalPurchaseAmount \* 100입니다. 전체 계좌 계산식은 보유 종목의 매입금액과 평가금액을 합산한
뒤, totalAssetAmount = cashBalance + stockEvaluationAmount, profitAmount = stockEvaluationAmount -
totalPurchaseAmount입니다.

HoldingResponse.stockId에는 holding.getId()가 아니라 holding.getStock().getId()가 들어가야 합니다. 명세의
stockId는 보유 주식 row ID가 아니라 종목 ID입니다.

Holding.stock은 LAZY이므로 holding.getStock()을 여러 번 호출하면 보유 종목 수만큼 추가 조회가 발생할 수 있습니
다. 이를 N+1 문제라고 합니다. 계좌 조회에서는 보유 종목의 Stock 정보가 반드시 필요하므로 HoldingRepository에
fetch join 쿼리를 추가하는 방식이 좋습니다.

@Query 안의 코드는 SQL이 아니라 JPQL입니다. select h from Holding h join fetch h.stock where h.account.id
= :accountId는 Holding 엔티티를 조회하되, 연관된 stock을 함께 즉시 로딩하고, Holding.account.id가 파라미터와
같은 것만 가져오라는 뜻입니다. @Param("accountId")는 메서드 파라미터와 JPQL의 :accountId를 연결합니다.

findAllByAccountIdWithStock이라는 메서드명 자체는 JPA 규칙이 아닙니다. @Query가 붙어 있으면 실제 동작은 메서드
명이 아니라 @Query 내용이 결정합니다. 이름은 사람이 읽기 좋게 붙인 설명용입니다. 이 메서드에서만 join fetch가
적용되고, 엔티티의 기본 LAZY 설정은 유지됩니다. 기본은 LAZY로 두고 필요한 조회에서만 fetch join을 쓰는 편이 좋
습니다.

findAllByAccount_Id의 언더바는 Spring Data JPA 파생 쿼리 메서드에서 중첩 속성을 명확히 구분하는 데 쓰입니다.
findAllByAccount_Id(Long accountId)는 Holding.account.id = accountId 조건으로 해석됩니다. 언더바 없이
findAllByAccountId도 경우에 따라 해석될 수 있지만, 엔티티에 accountId라는 직접 필드가 있을 수도 있으므로 언더
바가 더 명확합니다.

@Query가 있으면 메서드명보다 @Query가 우선입니다. @Query가 없으면 Spring Data JPA가 메서드명을 해석해서 쿼리를
생성합니다. 메서드명이 엔티티 필드 구조와 맞지 않으면 애플리케이션 시작 시점에 No property ... found 같은 오류
가 납니다. 단순 조건은 파생 메서드, fetch join이나 복잡한 조건은 @Query를 쓰는 것이 좋습니다.

인증 실패 시 response body가 없는 이유는 인증 실패가 컨트롤러 전에 Spring Security 필터 체인에서 발생하기 때문
입니다. GlobalExceptionHandler는 컨트롤러/서비스 예외를 주로 처리하고, Security 영역의 인증 실패는
AuthenticationEntryPoint, 권한 부족은 AccessDeniedHandler로 처리하는 것이 정석입니다. 잘못된 JWT가
Authorization 헤더에 들어온 경우는 JwtAuthenticationFilter 내부에서 JwtException 등을 catch해서 JSON 응답을 직
접 쓰는 방식이 필요할 수 있습니다.
