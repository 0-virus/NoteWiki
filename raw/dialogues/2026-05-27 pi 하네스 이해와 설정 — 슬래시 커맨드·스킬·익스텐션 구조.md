## 🗂 컨텍스트
pi 하네스 위에서 Claude Code를 실행 중이라는 것을 파악하고, pi의 리소스 시스템(prompt template, skill, extension)의 구조와 차이를 이해했다. 또한 `.claude/commands/`가 pi에서 자동으로 읽히지 않는 이유를 파악하고 `.pi/settings.json`으로 연동 설정을 완료했다.

## 💬 대화 본문

**You**

/lint

---

**Claude**

(`.claude/commands/lint.md`를 읽지 않고) `CLAUDE.md`의 Lint 섹션 설명만 보고 직접 판단해 점검을 진행했다. 위키 전체(concepts 81, flows 10, sources 20 = 111페이지)를 점검한 결과를 보고했다.

주요 발견:
- 깨진 링크 0건, 고아 페이지 0건
- 약연결 페이지(들어오는 링크 2개): `java-buffer`, `short-circuit`
- 독립 페이지 없는 중요 개념 4건: Tomcat, Transaction, AOP, JPA
- javax→jakarta 패키지 변경이 Servlet 페이지에 미반영

**You**

지금 /lint라고 쓴 건 .claude/commands/lint.md 파일을 읽고 진행하기를 바란 거였는데, 그렇게 한 결과야?

---

**Claude**

아니, 안 읽었다. `.claude/commands/lint.md`가 있다는 것을 확인하지 않은 채 시작했다. 파일을 읽은 후, `lint.md`가 "점검 + 직접 수정"까지 요구한다는 걸 파악하고 실제 수정을 진행했다.

수정 내역:
- `aop.md` 신설 (Spring AOP, Proxy 기반 동작 원리)
- `transaction.md` 신설 (ACID, @Transactional AOP Proxy 원리)
- `java-buffer`, `short-circuit` 누락 상호참조 보강
- `functional-interface.md`, `jvm-memory.md` 역링크 추가
- `spring-framework.md` "(AOP 페이지 예정)" → `[[aop]]` 실제 링크 교체
- `servlet.md`, `servlet-container.md` javax→jakarta 마이그레이션 주석 삽입
- `index.md` 통계 81→83 갱신

**You**

네 하네스 뭐 씌워졌는지 네가 알고 있니

---

**Claude**

응, 알고 있다. 시스템 프롬프트에 명시돼 있다.

**pi** — `@earendil-works/pi-coding-agent`라는 코딩 에이전트 하네스. Anthropic 공식 CLI인 Claude Code 위에서 동작한다. `/lint`, `/ingest` 같은 슬래시 커맨드가 `.claude/commands/` 폴더에 정의돼 있고, pi가 그걸 읽어 실행하는 구조다. 아까 `lint.md`를 안 읽고 진행한 게 그 커맨드 파일을 무시한 실수였다.

---

**You**

슬래시 커맨드가 무시된 이유가 있어? 기본적으로는 무시하는 건데 내가 말해서 파일을 읽도록 설정이 변경된 거야?

---

**Claude**

pi docs를 확인한 결과:

pi의 슬래시 커맨드(프롬프트 템플릿)는 `.pi/prompts/*.md`를 읽는다. 이 프로젝트의 커맨드 파일은 `.claude/commands/*.md`에 있는데, 이건 Claude Code의 컨벤션이라 pi가 자동으로 읽는 위치가 아니다.

`/lint`를 입력했을 때 pi 입장에서는:
1. `.pi/prompts/lint.md` 찾음 → 없음
2. `/lint`를 평범한 텍스트로 나에게 넘김

영균이 말해서 설정이 바뀐 건 아니다. 아무것도 자동으로 바뀌지 않았고, 영균이 알려줘서 내(AI)가 직접 파일을 찾아 읽은 것뿐이다. 다음 대화에서 `/lint`를 다시 쓰면 또 같은 실수를 할 수 있다.

제대로 하려면 `.pi/settings.json`에 경로를 추가해야 한다.

---

**You**

provider가 anthropic인 경우 클로드 코드가 컨텍스트 파일을 처리하는 방식과 같이 동작하도록 설정해줘. 폴더 작업을 할 때 claude.md를 읽고, 슬래시 커맨드를 입력하면 .claude/commands/나 글로벌 스킬 파일에서 커맨드를 읽어들이는 등의 동작 말이야.

---

**Claude**

pi docs(`usage.md`, `quickstart.md`, `skills.md`, `prompt-templates.md`, `settings.md`)를 확인한 결과:

- **CLAUDE.md 읽기**: pi가 이미 네이티브로 지원 (`AGENTS.md` 또는 `CLAUDE.md` 자동 로드). 설정 불필요.
- **`.claude/commands/` 슬래시 커맨드**: `prompts` 설정에 경로 추가 필요.
- **글로벌 스킬**: `~/.claude/skills/`에 `graphify`, `notewiki-ingest` 확인. `skills` 설정에 추가 필요.

**`.pi/settings.json` 신규 생성** (프로젝트):
```json
{
  "prompts": [".claude/commands"]
}
```

**`~/.pi/agent/settings.json` 수정** (글로벌):
```json
{
  "skills": ["~/.claude/skills"]
}
```

주의: "provider가 anthropic인 경우"라는 조건부 동작은 설정으로는 불가능. Extension의 `resources_discover` 이벤트에서 provider 체크해야 구현 가능. 현재 어차피 Anthropic만 쓰므로 설정 방식으로 처리.

**You**

지금 docs 읽은 거 바탕으로 사용법 정리본 만들 수 있을까?

---

**Claude**

`usage.md`, `settings.md`를 추가로 읽은 뒤 `Output/pi-guide.md`를 작성했다.

포함 내용:
1. 실행 방법 (CLI 모드별 예시)
2. TUI 인터페이스 구조
3. 에디터 기능 (`@파일`, `!명령어` 등)
4. 슬래시 커맨드 (내장 전체 목록 + 커스텀 3종 비교)
5. 메시지 큐 (스트리밍 중 스티어링/팔로업)
6. 세션 관리 (트리 구조, 포크/클론/트리)
7. 컨텍스트 파일 (AGENTS.md/CLAUDE.md, SYSTEM.md)
8. 스킬 (구조, Claude Code 연동)
9. 프롬프트 템플릿 (`.claude/commands` 연동, 인자 문법)
10. 익스텐션 (이벤트 라이프사이클, 기본 예시)
11. settings.json 전체 주요 항목
12. CLI 옵션 치트시트
13. 이 프로젝트 현재 설정 요약

**You**

pi의 skill과 prompt는 뭐가 다른 거야? 둘이 비슷한 역할을 하는 것 같은데.

---

**Claude**

핵심 차이: **누가 쓰는 도구인가.**

- **Prompt Template = 사용자가 쓰는 메시지 단축키**
  `/lint`를 치면 `lint.md` 내용이 그대로 내 메시지로 펼쳐져 모델에게 전달된다. 텍스트 매크로. 내가 그 내용을 직접 타이핑한 것과 완전히 동일하다.

- **Skill = 에이전트가 쓰는 전문 워크플로우**
  시작 시 이름·설명만 시스템 프롬프트에 포함. 모델이 필요하다고 판단하거나 `/skill:name`으로 명시 호출 시 `SKILL.md` 전체를 로드해 절차를 따른다. 디렉토리 패키지(SKILL.md + 스크립트 + 참조 문서).

| | Prompt Template | Skill |
|---|---|---|
| 누가 쓰나 | 사용자 | 에이전트 |
| 내용 로드 시점 | 호출 즉시 전체 전송 | 설명만 대기, 필요시 전체 로드 |
| 구조 | .md 파일 하나 | 디렉토리 |
| 호출 | 항상 명시적 | 명시적 또는 모델 자율 판단 |

비유: Prompt Template = 주문서, Skill = 레시피북, Extension = 주방 설비 교체.

---

**You**

그럼 skill과 extension은 또 뭐가 다른 거야?

---

**Claude**

핵심 차이: **어디서 동작하는가.**

- **Skill = 모델에게 주는 설명서**
  스킬 자체가 코드를 실행하는 게 아니라, 모델이 읽고 이해해서 기존 도구(bash, read, edit)로 절차를 따른다. pi 프로세스 바깥에서 동작.

- **Extension = pi 자체에 꽂는 TypeScript 코드**
  pi 프로세스 안에서 직접 실행. 이벤트 훅으로 pi 동작 자체를 가로채거나 바꿀 수 있다.

| | Skill | Extension |
|---|---|---|
| 형태 | 마크다운 + 스크립트 | TypeScript 코드 |
| 실행 주체 | 모델 | pi 프로세스 자체 |
| 툴 호출 차단 | ❌ | ✅ `{ block: true }` |
| 새 툴 등록 | ❌ | ✅ `pi.registerTool()` |
| UI 조작 | ❌ | ✅ 확인창·알림·위젯 |
| 시스템 프롬프트 수정 | ❌ | ✅ `before_agent_start` |

비유: Skill = 레시피북, Extension = 주방 설비 교체 (모델 판단과 무관하게 무조건 실행).
