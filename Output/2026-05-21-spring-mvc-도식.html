먼저, 머릿속 정리 — Spring MVC는 “네 가지 분리축”의 교차점

overview

Spring MVC를 처음 보면 컴포넌트가 너무 많아서 길을 잃기 쉽다. 하지만 그 모든 부품은 결국 네 가지 분리축 위에 놓여 있다. 도식을 하나씩 보기 전에 이 축들부터 머리에 박아두자.

분리축무엇을 나누나대표 부품왜?
가로채기 위치 스프링 밖 / 안 Filter / Interceptor 각자 다룰 수 있는 정보의 깊이가 다름
표현 분리 (MVC) 요청 처리 / 데이터 / 화면 Controller / Model / View 화면 변경과 로직 변경을 떨어뜨림
시스템 계층 (3-tier) 웹 / 비즈니스 / 데이터 @Controller / @Service / @Repository 각 계층이 알아야 할 게 좁아지면 변경이 안전해짐
객체 책임 운반 / 접근 / 도메인 DTO / DAO / Entity (DO) 도메인 변경이 클라이언트로 새지 않게 격리
핵심 통찰 — 이 네 축은 같은 일을 네 번 하는 게 아니라, 다른 차원의 분리다. 그래서 한 요청을 따라가면 네 축이 동시에 작동하는 게 보인다 (도식 1·3에서 확인).

02

도식 2. DispatcherServlet 내부 — Front Controller가 “위임”으로 일하는 법

spring-mvc-core

DispatcherServlet은 스프링 위에서 도는 단 하나의 Servlet이다. 모든 URL이 이 친구에게 먼저 들어온다. 그런데 정작 자기는 매핑 룰도, 호출 방식도, 뷰 렌더링도 직접 안 한다 — 전부 위임한다. 이 “위임 구조”가 Spring MVC가 그토록 유연한 이유.

4단계 위임

                           ┌──────────────────────────────────────┐
   요청 도착 ─────────────▶ │  ① HandlerMapping                    │
   /posts/42 GET            │     "URL과 메서드를 보고              │
                            │      어느 Handler(컨트롤러 메서드)에   │
                            │      넘길지 결정"                     │
                            └──────────────┬───────────────────────┘
                                           │ HandlerExecutionChain
                                           ▼ (Handler + Interceptor 목록)
                            ┌──────────────────────────────────────┐
                            │  ② HandlerAdapter                    │
                            │     "그 Handler를 어떻게 호출해야      │
                            │      하는지를 안다"                    │
                            │     (@Controller, HttpRequestHandler  │
                            │      등 다양한 핸들러 타입 대응)        │
                            └──────────────┬───────────────────────┘
                                           │ Interceptor.preHandle
                                           ▼
                            ┌──────────────────────────────────────┐
                            │     @Controller 메서드 실행           │
                            │     반환: ModelAndView 또는           │
                            │           @ResponseBody 객체          │
                            └──────────────┬───────────────────────┘
                                           │ Interceptor.postHandle
                                           ▼
                            ┌──────────────────────────────────────┐
                            │  ③ ViewResolver                      │
                            │     "뷰 이름('posts/detail')을        │
                            │      실제 JSP/Thymeleaf 파일로 매핑"   │
                            │     (@ResponseBody면 건너뜀)          │
                            └──────────────┬───────────────────────┘
                                           ▼
                            ┌──────────────────────────────────────┐
                            │  ④ HttpMessageConverter / View       │
                            │     · @ResponseBody → Jackson이 JSON  │
                            │     · 일반 View → 템플릿 렌더링         │
                            └──────────────┬───────────────────────┘
                                           │ Interceptor.afterCompletion
                                           ▼
                                       응답 송출

왜 “위임”인가

만약 DispatcherServlet이 매핑·호출·렌더링을 직접 다 한다면, “URL 패턴 바꾸자”나 “JSON으로 응답 형식 바꾸자” 한 마디에 DispatcherServlet 코드를 손대야 한다. 위임 구조 덕분에 우리는 해당 부품(예: HttpMessageConverter)만 교체하면 된다. OCP(Open-Closed)를 인프라 수준에서 실현한 셈.

“스프링이 없으면 우리가 직접 짜야 할 것”과의 매핑

순수 Servlet이라면DispatcherServlet이 대신해주는 것
@WebServlet("/posts/*")로 각 URL마다 Servlet 분리HandlerMapping이 한 번에 처리
doGet / doPost 분기, 파라미터 파싱HandlerAdapter가 메서드·파라미터 바인딩
request.setAttribute + forward()ViewResolver가 뷰 이름만으로 렌더
JSON 응답이면 PrintWriter로 직접 {...} 작성Jackson이 객체 → JSON 자동 변환
왜 DispatcherServlet은 “Front Controller”인가 — 모든 요청이 “정문 한 곳”으로 들어오기 때문. 반대 디자인(URL마다 Servlet을 따로 두는 것)은 공통 처리(인증·로깅·뷰 결정)를 매번 복붙해야 한다. 정문 하나로 모으면 그 모든 공통 처리를 한 곳에서 표준화할 수 있다.
03

도식 3. Spring MVC 구조 — MVC와 3-tier가 동시에 작동한다

two-axes

가장 자주 헷갈리는 지점. MVC와 3-tier는 같은 것이 아니다. MVC는 표현(presentation) 분리고, 3-tier는 시스템 계층 분리다. 축이 다르기 때문에 두 분리는 동시에 적용된다. 

                        ─── MVC 축 (표현 분리) ───
                  Controller        Model            View
                       ▼              ▼                ▼
   ┌───────────────────────────────────────────────────────────┐
   │  WEB LAYER                                                │  ┐
   │  @Controller / DispatcherServlet / Filter / Interceptor   │  │
   │  Thymeleaf · JSP · HttpMessageConverter                   │  │
   ├───────────────────────────────────────────────────────────┤  │ 3-tier 축
   │  BUSINESS LAYER                                           │  │ (시스템 계층)
   │  @Service · @Transactional · Domain Model(Entity의 행위)  │  │
   ├───────────────────────────────────────────────────────────┤  │
   │  DATA LAYER                                               │  │
   │  @Repository · DAO · JdbcTemplate · JPA · Entity 매핑     │  ┘
   └───────────────────────────────────────────────────────────┘
                                ▼
                              DATABASE

축이 만나는 지점

  • Controller — MVC의 C이자 3-tier의 Web Layer 진입점. 두 축이 겹쳐 보여서 헷갈린다.
  • Model — MVC에서는 “Controller가 View로 넘기는 데이터”. 3-tier에서는 “Domain Model(Service 계층의 핵심)”. 같은 단어가 두 뜻으로 쓰임.
  • View — MVC 전용 개념. 3-tier에는 따로 대응하는 게 없다 (Web Layer 안에 다 들어감).

한 표로

축분류 기준분할어디 산다
MVC 표현 분리 (UI) Controller · View · Model Web Layer 안
3-tier 시스템 계층 Web · Business · Data 시스템 전체에 걸침
외워두기 — “MVC는 화면 쪽만 분리, 3-tier는 화면 안쪽까지 포함한 전체 분리.” Spring에서 @Controller → @Service → @Repository로 이어지는 호출이 바로 3-tier. 그 안에서 Controller와 View가 협력하는 게 MVC.
04

도식 4. 데이터 객체 4종 — DTO / DAO / DO / Entity

data-objects

이름이 비슷해서 가장 잘 섞이는 4개. 결론부터 한 줄로:

요약 — DAO는 “행동(접근)”, 나머지 셋은 “객체”다. 그 중에서 Entity는 “식별자(id)를 가진 도메인 객체”, DO는 “도메인 모델에 속하는 객체 모두(Entity의 부모 카테고리)”, DTO는 “계층/시스템 경계를 건너는 운반체”다.

DTO

Data Transfer Object

계층 사이를 오가는 운반체

  • 데이터만 있고 행위는 없음
  • 요청/응답 시나리오마다 다른 모양
  • 주로 record나 Lombok DTO
  • 변환 위치: 보통 Service에서
필드 결정 기준 = 유스케이스 (엔티티 아님)

DAO

Data Access Object

DB 접근을 전담하는 객체

  • SQL/JDBC 호출을 한곳에 모음
  • Spring에서는 @Repository
  • JPA Repository 인터페이스도 이 자리
  • Service는 “무엇을”만, DAO는 “어떻게”를 안다
4종 중 유일하게 “행동” 객체다

Entity

Domain Entity

식별자(id)를 가진 도메인 본체

  • 상태 + 식별자 + 행위
  • JPA에서 @Entity 어노테이션
  • DB 테이블과 매핑되는 경우가 많음
  • 비즈니스 규칙을 자기 안에 품음
Order.cancel()처럼 도메인 규칙을 실행

DO

Domain Object

도메인 모델에 속하는 모든 객체

  • Entity의 상위 카테고리
  • VO(Value Object)도 DO의 일종
  • “DB와의 매핑” 의무 없음
  • 한국 실무에선 “Entity와 사실상 같은 뜻”으로도 씀
⚠️ 팀마다 정의가 다름 — 합의된 의미를 먼저 확인

객체 vs 행동 — 가장 큰 오해 풀기

                      [ Controller / Service ]
                            │
                            │  postDao.findById(42)
                            ▼
                    ┌──────────────────┐
                    │      DAO         │   ← 객체이긴 한데 본질은 "행동"(SQL 호출)
                    │   @Repository    │
                    └────────┬─────────┘
                             │  SELECT * FROM posts WHERE id = 42
                             ▼
                          [ DB ]
                             │
                             ▼  ResultSet
                    ┌──────────────────┐
                    │     Entity       │   ← DB 한 행을 채운 도메인 본체
                    │     (Post)       │
                    └────────┬─────────┘
                             │  Service에서 변환
                             ▼
                    ┌──────────────────┐
                    │       DTO        │   ← Controller가 클라이언트에 줄 운반체
                    │  PostResponseDto │
                    └──────────────────┘

한 표로 비교

구분DTODAOEntityDO
본질 객체(운반체) 행동(접근자) 객체(도메인 본체) 객체(도메인 모델 일반)
행위(메서드) 없음 findAll/save/... 있음 (규칙 실행) 있을 수 있음
식별자(id) 있어도 됨, 의미 약함 해당 없음 핵심 경우에 따라 (VO는 없음)
DB 매핑 아님 매핑을 수행함 대개 매핑됨 의무 없음
위치 계층 경계 Data Layer Business/Data Layer Business Layer
Spring 일반 클래스/record @Repository @Entity 특정 어노테이션 없음

DO와 Entity — 관계가 모호한 이유

엄밀히 말하면 Entity ⊂ DO다. 즉 DO가 더 큰 범주고, Entity는 그 중 “식별자로 구별되는 것”. DDD(Domain-Driven Design)에선 DO를 다시 Entity(식별 가능)와 Value Object(값으로만 구별, 예: Money(1000, KRW))로 나눈다.

그런데 한국 실무에서는 “DO”라는 용어를 거의 안 쓰거나, Entity와 사실상 동의어로 쓰는 경우가 많다. DDD 책이나 외국 코드베이스를 볼 때만 “DO ⊃ Entity”라는 위계가 의미를 가진다.

영균 실무에서 — 처음엔 “DTO ↔ Entity” 두 개로만 생각해도 충분. DO를 의식해야 할 때는 (a) DDD 책 읽을 때, (b) Value Object(예: Money, Address)를 도입할 때, (c) 외국 코드베이스 읽을 때 정도.

도식 5. 한 시나리오로 따라가기 — “POST /posts” 요청의 라이프

walkthrough

영균이 만드는 미니 SNS의 글 작성 요청. 위 도식들이 어떻게 한 줄로 꿰어지는지.

①
브라우저가 POST /posts + JSON {title, body} 송신 → Tomcat이 TCP 받고 HTTP 파싱.
②
Filter Chain 통과 — CharacterEncodingFilter가 UTF-8 강제, CORS 필터가 Origin 검사.
③
DispatcherServlet 진입. HandlerMapping이 “POST /posts → PostController.create()” 결정.
④
AuthInterceptor.preHandle — 세션에 로그인 정보 없으면 401 반환하고 컷. 있으면 통과.
⑤
HandlerAdapter가 Jackson을 시켜 요청 본문 JSON → PostCreateRequestDto로 역직렬화. 그걸 컨트롤러 메서드 파라미터에 주입.
⑥
@Controller 메서드 본체: postService.create(dto, loginUser) 호출. 컨트롤러는 흐름 조립만 한다.
⑦
@Service PostService.create: DTO를 Entity(new Post(...))로 변환, 도메인 규칙(예: post.validateTitleLength()) 적용, postRepository.save(post) 호출. @Transactional 경계도 여기.
⑧
@Repository = DAO가 INSERT SQL 실행. JPA면 EntityManager.persist, JdbcTemplate이면 update(...). DB에 한 행 박힘.
⑨
저장된 Entity를 받아 PostResponseDto로 다시 변환 (Service가 함). 민감 필드(예: 작성자의 비밀번호 해시)는 빠진다 — DTO의 차단 효과.
⑩
컨트롤러가 DTO 반환 → @ResponseBody이므로 ViewResolver는 건너뜀 → HttpMessageConverter(Jackson)가 DTO → JSON 직렬화.
⑪
Interceptor.postHandle / afterCompletion 실행 (감사 로그, 성능 측정).
⑫
응답이 Filter 역순 통과 (gzip 압축, 응답 로깅) → 브라우저가 JSON 수신.
오늘의 한 줄 — 한 요청은 Filter → Dispatcher → Interceptor → Controller → Service → Entity ↔ DAO → DB를 거쳐 역순으로 빠져나온다. 그 길 위에서 DTO는 양 끝(요청·응답)에서만 등장하고, 중심엔 늘 Entity가 있다.

부록. 자주 헷갈리는 짝

cheat-sheet
헷갈리는 짝이렇게 구분
MVC ↔ 3-tier MVC는 화면(presentation) 분리, 3-tier는 시스템 계층 분리. 축이 다름.
Filter ↔ Interceptor Filter는 스프링 밖(HTTP만 봄), Interceptor는 스프링 안(Bean·HandlerMethod 봄).
Servlet ↔ DispatcherServlet DispatcherServlet도 하나의 Servlet이다. 다만 모든 URL을 한 곳으로 받는 Front Controller.
Service ↔ Controller Controller는 “HTTP 입출력 조립”, Service는 “비즈니스 로직 실행 + 트랜잭션 경계”.
DTO ↔ Entity DTO는 경계의 계약(유스케이스 기반), Entity는 도메인의 본체(행위 포함).
DAO ↔ Repository 같은 자리. Spring에선 @Repository가 DAO의 자리. JPA Repository 인터페이스도 여기.
DO ↔ Entity 엄밀히는 DO ⊃ Entity (DO 안에 Entity와 Value Object가 있음). 실무에선 거의 동의어로 쓰임.
HandlerMapping ↔ HandlerAdapter Mapping은 “누가 처리할지 찾기”, Adapter는 “찾은 그놈을 실제로 호출하기”.
ViewResolver ↔ HttpMessageConverter 전자는 뷰 이름 → 템플릿 파일, 후자는 객체 ↔ JSON/XML 변환. @ResponseBody면 후자만.
