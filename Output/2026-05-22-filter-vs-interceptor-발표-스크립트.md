**왼쪽**, "이 사용자, 로그인은 했어?" (로그인 체크)

그래서 우리한테 필요한 건, **"한 곳에서 가로채서 한 번에 처리"**해 주는 무언가예요.
바로 그 "가로채는 자"가 오늘 주인공인 **필터**와 **인터셉터**입니다. ( ▶ )

## 🎬 슬라이드 5 — Filter의 위치 (1:40 ~ 2:35)

> 화면: 방문객 → 🛡️ Filter → 아파트 단지

이해를 돕기 위해 비유를 들어보겠습니다.

우리가 만든 웹 서버를 한 채의 아파트 단지라고 하면, 사용자의 HTTP 요청은 **방문객**이고, 우리가 짠 컨트롤러는 **702호에 사는 입주민**이에요.

**필터는 단지 정문의 경비원**입니다.
방문객이 단지 정문에 도착했을 때 가장 먼저 만나는 사람이죠.

여기서 중요한 게 하나 있어요.
필터는 **스프링이 아니라 톰캣** — 즉 **서블릿 컨테이너 영역**에서 동작합니다.
스프링 건물에 들어가기 전, 단지 정문에서 일하는 거예요.

그래서 필터는 자바 서블릿이 처음 나왔을 때부터 있던 기술이고, 스프링이 없어도 그냥 존재합니다.
지난번에 MVC 발표 때 봤던 그 **서블릿 스펙**, 거기 들어 있는 표준 기술이에요. ( ▶ )

## 🎬 슬라이드 6 — Filter가 하는 일 (2:35 ~ 3:25)

> 화면: 인코딩 / XSS·CORS / 로깅 카드 3장

그럼 정문 경비원은 무슨 일을 할까요?

**첫째**, "마스크 안 쓰셨으면 못 들어가요" — **인코딩 설정**입니다.
모든 요청을 UTF-8로 강제하는 거죠. 그래서 한글이 안 깨집니다.

**둘째**, "수상한 물건은 빼고 들어가세요" — **XSS, CORS 같은 웹 보안**입니다.
악성 스크립트 차단이나 다른 도메인 접근 정책 같은 거요.

**셋째**, "방문 기록은 남깁니다" — **공통 로깅**이에요.
누가, 언제, 어떤 요청을 보냈는지 단지 전체의 출입 기록을 남기는 거죠.

여기서 핵심은요, 이 모든 일이 **HTTP 요청·응답 그 자체를 가공**하는 일이라는 점이에요.
"이 요청이 어느 컨트롤러로 갈지", "이 사용자가 누군지" 같은 건 필터는 모릅니다.
**스프링을 거치기 전 단계**니까요. 단지 전체의 안전만 관리해요. ( ▶ )

## 🎬 슬라이드 8 — Interceptor의 위치 (3:35 ~ 4:30)

> 화면: DispatcherServlet → 🕴️ Interceptor → Controller

같은 아파트 비유를 이어가 볼게요.

방문객이 정문 경비(필터)를 통과해서 단지 안으로 들어왔어요. 그러면 1층 로비에 안내 데스크가 있죠.
이 안내 데스크가 바로 스프링의 **디스패처 서블릿**입니다. "702호 가시려고요? 엘리베이터는 저쪽이에요" 하고 길을 알려 줘요.

그렇게 엘리베이터를 타고 702호 앞에 도착했을 때 — 초인종을 누르기 **딱 직전**에, 그 문 앞을 지키고 있는 사람이 있어요.
**개인 비서**, 또는 **사설 경호원**이라고 생각하시면 돼요. 이게 바로 **인터셉터**입니다.

- "입주민 연락처나 초대장 있으세요?" — **로그인 세션 확인**
- "어, 이 분 블랙리스트인데요. 돌려보내겠습니다." — **권한 체크**

여기가 결정적으로 다른 점이에요.
필터는 **단지 정문**에 있어서 입주민 사정을 모르지만, 인터셉터는 **입주민 바로 코앞**에서 일해요.
그래서 입주민의 사정 — 즉 **스프링의 빈(Bean)이나 데이터**에 자유롭게 접근할 수 있습니다. ( ▶ )

## 🎬 슬라이드 9 — Interceptor가 하는 일 (4:30 ~ 5:20)

> 화면: 세션 / 권한 / 감사 카드 3장

비서가 구체적으로 뭘 하느냐.

**첫째**, **로그인 세션 체크**.
"`HttpSession`에 로그인 정보가 있나?" 보고, 없으면 로그인 페이지로 돌려보내죠.

**둘째**, **권한(Role) 검사**.
이건 좀 멋있어요. 인터셉터는 "이 요청이 **어느 컨트롤러 메서드**로 가는지"를 알고 있거든요.
그래서 `@AdminOnly` 같은 어노테이션이 메서드에 붙어 있는지까지 검사할 수 있어요.

**셋째**, **컨트롤러별 감사·성능 측정**.
어떤 사용자가 어떤 핸들러에 들어왔는지, 실행에 몇 ms 걸렸는지 다 잴 수 있죠.

핵심은요, 인터셉터는 **스프링 영역 안**에 살아요.
그래서 우리가 지난 시간에 봤던 **DI**, `@Autowired`로 `UserService` 같은 빈을 그냥 받아서 쓸 수 있어요.
필터에서는 이게 좀 어색해요. 왜 어색한지는 곧 보여 드릴게요. ( ▶ )

## 🎬 슬라이드 11 — 본질적 질문 (5:30 ~ 6:50)

> 화면: Filter(거친 그물) / Interceptor(정교한 손길) 두 카드

제가 처음에 진짜 궁금했던 게 이거였어요.

> "**필터만으로도 다 할 수 있는 거 아닌가요?** 왜 굳이 인터셉터를 또 만들었지?"

답부터 말씀드리면, **할 수는 있어요. 그런데 어색해요.**
왜 어색한지가 오늘 발표의 핵심입니다.

**왼쪽**, 필터를 봐주세요.
필터는 단지 전체를 지키는 **거친 그물**이에요. 그래서 다룰 수 있는 게 `HttpServletRequest`와 `Response`, **딱 HTTP 그 자체**예요.
스프링 빈에 손이 안 닿아요. `@Service`로 만든 `UserService`를 필터에서 쓰려고 하면 `FilterRegistrationBean` 같은 우회 등록이 필요해요.
그리고 결정적으로, 필터에서 예외가 터지면 — **톰캣의 그 지저분한 흰 화면**이 사용자한테 그대로 뜹니다.

**오른쪽**, 인터셉터.
인터셉터는 입주민 코앞에서 일해요. 그래서 `HandlerMethod` — 즉 "어느 컨트롤러의 어느 메서드로 가는지"까지 알아요.
스프링 안에서 살기 때문에 `@Autowired`로 `UserService` 그냥 받으면 끝이고요.
예외가 터져도 `@ControllerAdvice`가 받아서 깔끔한 JSON이나 에러 페이지로 바꿔 줄 수 있어요.

그래서 정리하면 — **거친 그물로 단지 전체를 막고, 정교한 손길로 호수마다 다르게 검사**하는 거예요.
같은 일을 두 명이 하는 게 아닙니다. **다른 깊이로, 다른 일**을 하는 거예요. ( 잠시 멈춤 ▶ )

## 🎬 슬라이드 12 — 예외 처리의 차이 (6:50 ~ 8:00)

> 화면: Filter의 에러 vs Interceptor의 에러

방금 잠깐 언급한 **예외 처리** 얘기를, 이게 진짜 중요한 차이라서 좀 더 풀어 볼게요.

**왼쪽, 필터에서 예외가 터지면?**
스프링의 `@ControllerAdvice` 있죠? 우리가 예외를 예쁘게 처리해 주는 그것. — 이게 **닿지 않아요.**
이유가 뭐냐면, 필터에서 터진 시점은 **아직 스프링 영역에 들어오기도 전**이거든요. 정문에서 일이 터졌으니까 스프링이 알 수가 없죠.
결과는 — 톰캣 기본 에러 페이지가 사용자에게 그대로 노출됩니다.

**오른쪽, 인터셉터에서 예외가 터지면?**
이미 스프링 안이에요. 그러면 `@ControllerAdvice`가 받아서 — 우리가 정의한 깔끔한 에러 JSON이나 에러 페이지로 돌려보낼 수 있죠.

이 차이가 왜 중요하냐면 — 이게 바로 **"같은 일을 어디서 하느냐가 왜 중요한가"**에 대한 답이에요.
어디서 일하느냐가 그 사람이 무엇을 할 수 있는지를 결정해요.
필터는 스프링 밖이라서 스프링의 좋은 인프라(`@ControllerAdvice`, 빈 주입)를 못 빌려 써요.
인터셉터는 스프링 안이라서 그걸 다 빌려 쓸 수 있고요.

그래서 실무 팁 하나 — **비즈니스 로직에 가까운 검사일수록 인터셉터로** 옮깁니다.
사용자한테 보여줄 에러 메시지를 깔끔히 제어할 수 있으니까요. ( ▶ )

## 🎬 슬라이드 14 — 선택 기준 (8:10 ~ 9:35)

> 화면: 흐름 다이어그램 + 비교표

화면 위쪽이 한 요청이 흘러가는 **전체 경로**예요.

> 사용자 → **Filter** → DispatcherServlet → **Interceptor** → Controller

한 번 요청이 단지에 들어와서 입주민을 만날 때까지, 두 명의 문지기를 차례로 통과합니다.

아래 표가 둘의 차이를 한눈에 정리한 건데, 한두 줄만 짚을게요.

- **위치**: 필터는 톰캣, 인터셉터는 디스패처와 컨트롤러 사이.
- **속한 세계**: 필터는 서블릿 스펙(스프링 밖), 인터셉터는 스프링 MVC(스프링 안).
- **다룰 수 있는 정보**: 필터는 HTTP 본체만, 인터셉터는 `HandlerMethod`까지.
- **빈 주입과 예외 처리**: 인터셉터가 압도적으로 자유로움.

그래서 선택 기준은 — 한 줄로 줄이면 이거예요.

> **거대한 작업은 필터, 정교한 작업은 인터셉터.**

거대한 작업이 뭐냐면, "모든 요청에 적용되는 인코딩", "XSS·CORS 같은 웹 보안", "모든 요청을 빠짐없이 남기는 로깅". — 이런 건 필터.
정교한 작업, 그러니까 "로그인 인증", "어드민 권한 제어", "특정 컨트롤러 감사" — 이런 건 인터셉터.

여러분이 나중에 코드 짤 때 "이거 필터에 둬야 하나, 인터셉터에 둬야 하나" 헷갈리면, 딱 하나만 물어보세요.

> **"이거, 스프링 빈을 써야 하나? `@ControllerAdvice`로 처리하고 싶나?"**
> 그러면 인터셉터. 아니면 필터.

이게 오늘 가져가실 결론입니다. ( ▶ )
