Filter

아파트 단지 정문의 경비원

서블릿 컨테이너(Tomcat)의 최전방. 스프링 건물에 들어가기 전에 방문객을 검사한다.

방문객
HTTP 요청
Filter
정문 경비원
아파트 단지
스프링 영역
필터는 서블릿 스펙의 일부다. → 스프링이 아니라 Tomcat 차원에서 동작한다.

필터는 무엇을 검사할까?

인코딩 설정

"마스크 안 쓰셨으면 못 들어가요." — 모든 요청에 UTF-8을 강제.

XSS / CORS

"수상한 물건은 빼고 들어가세요." — 공통 보안 필터링.

"방문 기록은 남깁니다." — 단지 전체의 접근 로그.

순수한 HTTP 요청/응답 자체를 가공하는 데 특화되어 있다.

Interceptor

입주민(컨트롤러) 문 앞의 비서

디스패처 서블릿(스프링 로비)과 컨트롤러(입주민) 사이에서 동작한다.

DispatcherServlet
스프링 로비
Interceptor
문 앞 비서
Controller
702호 입주민
초인종 누르기 직전 문 앞을 지키고 있는 사설 경호원.

인터셉터는 무엇을 검사할까?

로그인 세션 체크

"입주민 연락처나 초대장 있으신가요?" — 세션을 들여다본다.

권한(Role) 검사

"이분 블랙리스트인데 돌려보낼게요." — 컨트롤러별 정교한 제어.

컨트롤러별 감사

어떤 핸들러에 진입했는지, 실행에 얼마나 걸렸는지 측정.

스프링의 영역 안에서 돌기 때문에, Bean 주입(@Autowired) 등 스프링 기능을 자유롭게 활용할 수 있다.
(입주민의 사정을 잘 아는 비서.)
— 04 —

Filter vs Interceptor

왜 두 명이나 필요한 걸까?

"필터만으로 다 할 수 있는 거 아닌가요?"

Filter — 거친 그물

단지 전체를 지킨다. HttpServletRequest/Response만 안다.

스프링 Bean에 손이 닿지 않는다. @Service 주입? 불가능.

예외가 나면 — 톰캣의 지저분한 흰 화면이 뜬다.

Interceptor — 정교한 손길

입주민(컨트롤러) 코앞에서 일한다. HandlerMethod를 안다.

스프링 안에서 산다. @Autowired UserService 가능.

예외도 @ControllerAdvice로 우아하게 받아낼 수 있다.

거친 그물(Filter)로 단지 전체를 막고, 정교한 손길(Interceptor)로 호수마다 다르게 검사한다.
"같은 일을 두 명이 하는 게 아니라, 다른 깊이로 다른 일을 한다."

에러가 터졌을 때 — 누가 받아주는가?

Filter에서 터지면?

스프링의 @ControllerAdvice가 못 잡는다.

아직 스프링 영역에 들어오지 않았기 때문.

톰캣 기본 에러 페이지가 그대로 노출

Interceptor에서 터지면?

이미 스프링 안. @ControllerAdvice의 손이 닿는다.

예쁜 에러 화면이나 JSON 응답으로 제어 가능.

사용자에게 깔끔한 응답

그래서 비즈니스 로직과 가까운 검사일수록 인터셉터로 옮긴다.
스프링이 마련해 둔 예외 처리 인프라를 그대로 쓸 수 있기 때문.
